# DESIGN.md — grug domain design notes (deep-dive companion to CONTEXT.md)

The vocabulary used in `services/`, `infra/`, and `web/`. Terms map to identifiers a contributor will encounter while reading code. New contributors should be able to read this file and use the terms correctly without grepping. Drift between this file and the code is a bug; fix the file in the same PR that renames a concept.

> **Architecture decisions live in [`docs/adr/`](docs/adr/).** This file is the lexicon; the ADRs are the load-bearing choices.

## Product surface

| Term | Definition |
|---|---|
| **Grug** | The bot. A hosted GitHub App at `grug.lol` that gates pull requests against a set of process checks. Open-source AGPL-3.0; self-host path in [`docs/SELF_HOST.md`](docs/SELF_HOST.md). |
| **Grug Boss** | Public-facing name of the GitHub App users install on their repos. Same product as Grug; "Grug Boss" is the GitHub Marketplace listing handle. |
| **Persona** | A bounded behavior surface Grug applies per PR. Two ship: **Chief/TPM** (static Definition-of-Ready check) and **Elder/code_reviewer** (LLM review + the deterministic security suite). Roadmap (epic #464): Guard, Smasher, Omen, Warder, Pulse, and the LLM scope-review half of TPM. Personas are per-repo togglable. |
| **TPM persona** | The process-gate persona (canonical name Chief). Static **Definition-of-Ready (DoR) check** that blocks merge if PR body is malformed. The companion **Scope review** half is wired as a `poolside_client.py` hook in `evaluate()` per the `__init__.py` docstring but is NOT shipped — see "Roadmap" rows. See [`services/{api,webhook}/personas/tpm/`](services/api/personas/tpm/). |
| **Elder persona** | The code-reviewer persona. Reads the PR diff and posts inline review comments + a check-run summary. Runs alongside TPM on every `pull_request` event (`opened`, `synchronize`, `ready_for_review`, `reopened`). The two personas dispatch INDEPENDENTLY — one failing does not skip the other. v1 modules: `personas/code_reviewer/diff_parser.py` (unified-diff → `DiffHunk`s) + `personas/code_reviewer/persona.py` (`evaluate_diff` rollup) + `personas/code_reviewer/dispatch.py` (orchestration — fetch diff, run LLM, publish via reviews+checks clients). LLM-driven via `llm_client.py` (Poolside + OpenRouter round-robin). Advisory by default (`code_reviewer_blocking=False` → check-run `conclusion=neutral`, review `event=COMMENT`); blocking flip via dashboard toggle. See specs 0015 + 0016 + [`services/{api,webhook}/personas/code_reviewer/`](services/api/personas/code_reviewer/). |
| **Pulse (roadmap)** | Scheduled (non-PR-triggered) persona work, named `stuck-PR-pulse` in the future-persona list in `services/webhook/main.py`. Not implemented. Intent: weekly issue-grooming sweep against Grugboard. |

## Process-gate concepts

| Term | Definition |
|---|---|
| **Definition of Ready (DoR)** | The standard a PR description must meet before it can merge. Enforced as a set of `CheckResult`s combined into a single GitHub check-run named `Grug — Definition of Ready`. |
| **DoR check** | Individual rule: `why`, `acceptance`, `estimate`, `scope-fence`, `issue-link`. Defined in [`services/{api,webhook}/personas/tpm/dor_checks.py`](services/api/personas/tpm/dor_checks.py) (mirrored — see ADR-0001). The five rule names are the prose label — there is no `DoRCheck` class; rules are functions returning `CheckResult`. |
| **CheckResult** | Outcome of one `DoR check` against one PR body. Frozen dataclass — fields `name: str`, `passed: bool`, `detail: str`. Pass/fail only — no third "warn" state. |
| **TpmEvaluation** | Aggregate result of running all `DoR check`s against one PR. Frozen dataclass returned by `evaluate_pull_request(...)` in `personas/tpm/persona.py`. Composes into a `CheckRunResult` for GitHub. |
| **CheckRunResult** | Frozen dataclass that maps directly onto GitHub's Checks API `POST /repos/{owner}/{repo}/check-runs` payload. Carries the `status=completed` ↔ `conclusion` cross-field invariant — enforced in `__post_init__`. See [`services/{api,webhook}/github_checks_client.py`](services/api/github_checks_client.py). |
| **`post_check_run` (publisher)** | Module-level function in `github_checks_client.py` that POSTs a `CheckRunResult` to GitHub. The acceptance-criteria spelling "CheckRunPublisher" is the *concept name* — the actual identifier is a function, not a class. |
| **Scope review (roadmap)** | Advisory LLM pass over PR title + body. Wired as a `poolside_client.py` hook called from `evaluate_pull_request(...)` per the `personas/tpm/__init__.py` docstring, but **`poolside_client.py` does not exist in the repo today** — feature is roadmap-only. Intended behavior: flag title↔body mismatch, AC testability, scope-creep, XL inflation; posted as a comment, never blocking. |

## Elder persona — code review

| Term | Definition |
|---|---|
| **Async Elder offload (#272/#368)** | The Elder LLM path runs **off** the synchronous webhook ACK. `receive_github_webhook` verifies HMAC, runs TPM (fast, no LLM) inline, then enqueues the Elder review via `async_dispatch.enqueue_elder_review` and returns `result="queued"` — GitHub is ACKed in <10s regardless of LLM latency. On the k8s runtime (`GRUG_K8S_RUNTIME`) the job runs in-process on a **daemon thread** (`_spawn_local_elder`); a pod restart mid-review drops the in-flight review (best-effort contract — a lost review re-triggers on the next push, or via the rerun queue). `run_elder_job` is **idempotent twice over**: on `delivery_id` (`install_store.claim_delivery`, win-once claim) AND on head SHA (`install_store.claim_review`, #397) so redeliveries and same-SHA re-triggers never double-post. It never re-raises (failures degrade to a published neutral check). The ACK path's remaining sync calls (SSM secret fetch, dispatch store work) are offloaded with `asyncio.to_thread` (#371) so concurrent deliveries do not serialize. Enqueue failure logs `elder_enqueue_failed` + returns `result="enqueue_failed"` — NO sync fallback, preserving the ACK guarantee. |
| **k8s execution paths (#368)** | Off-Lambda the same two async surfaces exist with k8s-native mechanics. **Elder:** `enqueue_elder_review` detects `GRUG_K8S_RUNTIME` (set only in the pod manifests) and runs `run_elder_job` on an in-process **daemon thread** - a pod has no invoke budget to escape, so the ACK handler returns immediately while the thread runs with the pod's lifetime. Idempotency + never-raise live in `run_elder_job`, unchanged. **Trade-off (decided):** a pod restart mid-review drops the in-flight review - the SAME best-effort contract as the Lambda async invoke; rejected SQS+worker for the hot path (new queue + DLQ + IAM + consumer for semantics we already accept). **Self-recovery (#418):** an UNHANDLED-error drop in `run_elder_job` (the class that paged P2 on 2026-06-16 - a list-shaped LLM response) no longer waits for a human re-push: it enqueues ONE durable re-run to `grug-rerun-jobs` (webhook-side `rerun.enqueue_rerun`), so the consumer re-runs it with the SQS redrive contract (visibility-timeout retries → DLQ after maxReceiveCount). Bounded: the rerun consumer calls `dispatch_code_review` directly (never `run_elder_job`), so it redrives on failure but never re-enqueues - at most one enqueue per drop, no loop. Best-effort: an enqueue failure (or a slim payload missing install/repo/pr ids) is logged, never raised. (A pod-restart-mid-review drop is still best-effort - there's no in-flight job to enqueue from - but that re-triggers on next push as before.) **Queues:** `consumer.py` (new entrypoint, same webhook image, Deployment `grug-consumer`, 1 replica) long-polls `grug-rerun-jobs.fifo` + `grug-cave-results.fifo` one thread each and feeds each message to the SAME handler the ESM used, in the SAME `Records` batch-of-1 shape - handlers and their tests untouched. Delivery semantics preserved: a rerun handler raise leaves the message (visibility-timeout redrive -> DLQ after 3), cave-results always deletes. A dead consumer thread kills the process (watchdog) -> kubelet restart. **IAM:** the `grug-k8s-pod` user already held the three queue grants; #368 adds write-only `s3:PutObject` on the cave-diffs `diffs/*` prefix (Elder spillover now happens on-pod). |
| **Every-commit review / per-head-SHA idempotency (#397)** | Elder reviews EVERY pushed commit on an open PR, exactly once per head SHA. The dispatcher already enqueues Elder on `synchronize` (every push) — verified live on PR #391, where all three commits each got a `Grug — Code Review` check-run (the "skipping" the issue reported was `gh pr checks` rendering a **`neutral`** advisory verdict, NOT a literal skip). The real gap was the inverse: idempotency was keyed only on `delivery_id`, so a non-push event at the SAME head SHA (`edited` on the PR body, `ready_for_review`) re-ran a full LLM review of unchanged code — duplicate reviews, wasted spend, noisy feed. **Fix:** `run_elder_job` now also claims per head SHA via `install_store.claim_review` (atomic `INSERT .. ON CONFLICT` win-once on `PK=REVIEW#<install>:<repo>:<pr>:<persona>:<head_sha>`, same TTL-takeover shape as `claim_delivery`). So every NEW head SHA reviews (no prior claim), and any same-SHA re-trigger — redelivery, `edited`, `ready_for_review` — claims-and-skips. The pre-existing `claim_delivery` gate stays (cheap exact-redelivery / #407-replay dedup, runs first); `claim_review` is the head-SHA layer on top. The claim is on ATTEMPT: an `errored` review still heals via the rerun queue (#305/#418 → `dispatch_code_review` directly), which bypasses this webhook-path gate, and an explicit `/rerun` (#87) likewise bypasses it — so neither idempotency layer wedges a legitimate re-review. (Only if BOTH the original dispatch AND the #418 self-recovery enqueue fail does the on-attempt claim block a same-SHA webhook re-trigger; recovery is then the DLQ terminal signal or an explicit `/rerun`, never worse than the pre-#397 "dropped, awaits re-push" terminal set.) Fails OPEN on a missing head SHA (a possible double-review beats a silent skip). |
| **SAST benchmark / eval harness (#399)** | A committed, re-runnable corpus + harness that measures Elder's vuln-detection recall + precision per backend - the acceptance gate for the SAST detection slices (#400/#401) and a standing regression guard (ADR-0006). Lives in `services/webhook/sast_benchmark/` (webhook-only tool, NOT mirrored). **Corpus** (`corpus.py`): one inline diff sample per canonical vuln class (cleartext-secret-log, hardcoded-credential, SQL/command/template-injection, SSRF, path-traversal, unsafe-deser, weak-crypto) each with ground truth `(vuln_class, path, is_true_positive)`, PLUS the known FALSE-POSITIVE shape (the #391 public-config-path log) that must NOT be flagged. **Pure core** (`scoring.py` - CI-safe, no LLM): `score(samples, findings_by_sample) -> BenchmarkReport` (per-class recall = TP samples flagged; precision = TP-flags / all-flags, where any finding on an FP sample is a false positive) + `compare_to_baseline(report, baseline)` (a per-class recall DROP vs the committed baseline is a regression). Unit-tested with SYNTHETIC findings - zero credits, deterministic. **Live runner** (`runner.py` + `backends.py`): backend-pluggable - reuses Elder's EXACT prompt (`llm_client._build_messages`) + parser (`_parse_response`) so it measures Elder, over a thin OpenAI-compatible transport so each backend (OpenRouter, Poolside, AND sparkles/the-Cave Ollama) is driven independently and scored per-backend. Backend endpoints/keys/models are ALL config-from-env (the Cave/sparkles endpoint is a tailnet address - NEVER a literal in this public repo; supplied via secret at run time). **Baseline** (`baseline.json`): recorded by an on-demand `workflow_dispatch` CI job (`benchmark.sast.yml`) that joins the tailnet (for sparkles) + reads the free-key secrets - never per-PR (no credit burn, no flake in the gate suite). No fabricated verdicts: recall/precision come from real backend runs (AC4). |
| **SAST detection tracer - clear-text-secret-log (#400)** | The first real SAST detection slice (ADR-0006 Option C), end-to-end for ONE class to prove the pipeline before broadening (#401). `personas/code_reviewer/sast.py` (mirrored): **recall layer** `scan_candidates(hunks) -> tuple[Candidate, ...]` - a PURE builtin detector (a log/print SINK + a secret-ish token on an added line; line numbers computed from the hunk so candidates are diff-anchored). Deliberately LIBERAL: it flags BOTH a real secret-in-log AND the #391 public-config-path log - discriminating them is the judge's job, not the detector's (a detector miss is unrecoverable; an over-flag is recovered by the judge). Per ADR-0006 the heavy Semgrep OSS engine slots in behind this same `scan_candidates` boundary at #401; #400 ships a lean builtin detector to keep the pod footprint flat. **Precision layer** `judge_candidates(...)` - reuses the EXISTING `llm_client.judge_findings` (is_real_bug) as the exploitability gate: candidate -> provisional finding -> judge; KEEP where `is_real_bug` (its `reasoning` becomes the source->sink rationale), SUPPRESS otherwise (the #391 log is dropped with a reason). FAIL-CLOSED on a judge outage / count mismatch (un-triaged candidates are suppressed, never posted raw - the noisy-SAST failure precision prevents). Kept candidates become ordinary `Finding`s; `persona.with_extra_findings` merges them into the evaluation and re-derives the conclusion by the SAME rule `evaluate_diff` uses (high/critical -> failure), so SAST findings flow through the EXISTING anti-hallucination + advisory/blocking publish path - no parallel posting. `dispatch_code_review` runs scan -> judge -> merge after `evaluate_diff`, best-effort (any SAST failure is additive and never breaks the core review). Advisory-by-default; the `code_reviewer_blocking` toggle makes a confirmed leak a merge gate. |
| **SAST full-class coverage - Semgrep engine (#401)** | Broadens detection from the one tracer class to the full set (SQL/command/template injection, SSRF, path-traversal, unsafe-deser, weak-crypto, XXE, hardcoded-credential, cleartext-secret-log) by swapping the ADR-0006 engine in behind the `scan_candidates` boundary #400 built - the ONLY change vs #400, since the judge + `with_extra_findings` merge + publish are engine-agnostic. `scan_candidates(hunks, file_contents, engine=None)` dispatches on `GRUG_SAST_ENGINE` (config value, vendor-neutral): `semgrep` = `scan_semgrep` (the OSS engine), default `builtin` = the #400 zero-dep detector (safe degrade). **scan_semgrep:** writes the changed files (`file_contents`, head-SHA content from #336) to a temp dir, runs `semgrep scan --config services/webhook/sast_rules/ --json` as a CLI subprocess (NO registry network - the rules are VENDORED offline in `sast_rules/python.yml`, one rule per class carrying `metadata.vuln_class`), maps each result to a `Candidate`, and keeps ONLY findings on lines THIS PR added (`DiffHunk.new_lines` per file - a PR review flags what the PR introduces, not pre-existing code). AC5 cost-bound: a per-review byte budget caps what Semgrep scans; over-budget files are skipped + logged (never silent). Best-effort: a missing `semgrep` binary / non-zero exit / timeout / unparseable output returns `()` + logs (SAST is additive). The semgrep CLI ships in the webhook image (`requirements.txt`); `GRUG_SAST_ENGINE=semgrep` is set on the webhook + consumer deployments (the review workloads); the mirrored api copy shells out rather than importing semgrep, so it needs neither the dep nor the rules. NOTE (corpus fix): SAST engines allowlist the canonical AWS docs example key, so the #399 hardcoded-credential corpus sample was changed to a realistic non-allowlisted value (else it would measure 0 recall for that class). |
| **SCA - dependency-CVE detection (#434, ADR-0007 Track 1)** | First Track-1 slice: flag dependencies a PR INTRODUCES/bumps-to that have known advisories, as a NEW candidate SOURCE feeding the SAME exploitability-judge + publish pipeline (the engine boundary generalizes from "scan code" to "scan candidates"). `sca.py` (mirrored): `extract_changed_deps(hunks)` (PURE) parses pinned `name==version` deps on ADDED lines of dependency manifests (requirements*.txt / pyproject / constraints; Python-first, multi-ecosystem deferred), diff-scoped + deduped + capped (AC5); `scan_dependencies(hunks)` queries the free public OSV advisory DB and emits a `Candidate(vuln_class="vulnerable-dependency", file, line, snippet)` per changed dep with advisories. **Engine note:** queries the OSV batch API DIRECTLY over httpx, NOT the pip-audit CLI - pip-audit's requirements-audit mode spins a venv + ensurepip that fails on the pod's readOnlyRootFilesystem (proven: SIGABRT); the direct OSV query is the same advisory data, pod-safe, ONE HTTPS call over the already-allowed 443 egress (free, no key, no SaaS spend, NO new dependency - httpx is already present). `dispatch_code_review` concatenates SAST + SCA candidates into ONE `judge_candidates` pass, then `with_extra_findings` merges the kept ones - so a vulnerable dep is judged for real reachability/impact and published advisory-by-default via the existing path. Best-effort throughout (OSV-unreachable -> () + log). The live OSV test is opt-in (`GRUG_SCA_LIVE_TEST`) so the per-PR gate stays deterministic; mocked tests cover the parse/judge/publish wiring. |
| **Secret/credential scanning (#436, ADR-0007 Track 1)** | Second Track-1 slice: flag committed secrets a PR INTRODUCES on added lines of ANY file type, as a NEW candidate SOURCE feeding the SAME exploitability-judge + publish pipeline. Closes the gap the SAST engine leaves: its vendored ruleset is `languages: [python]`, so a secret committed to a `.env` / YAML / shell / JS / Dockerfile is invisible to it. `secret_scan.py` (mirrored): `scan_secrets(hunks) -> tuple[Candidate, ...]` is a PURE, file-type-agnostic detector over ADDED lines only (diff-scoped, content-deduped by the exact matched credential per `(file, kind)`, capped at `_MAX_SECRETS=100` and a `_MAX_SCAN_BYTES` aggregate work budget for cost - AC6). Two recall rules: (A) high-signal provider token patterns (AWS access-key id, GitHub / Slack / Google / Stripe tokens, PEM private-key header) matched anywhere on the line, no key-name context needed; (B) a generic secret-ish assignment (`secret`/`token`/`api_key`/`password`/... assigned with `=` or `:`) whose value clears a Shannon-entropy gate (length + bits/char), so low-entropy placeholders like `your-key-here` never become candidates. Emits `Candidate(vuln_class="exposed-secret", file, line, snippet)`; `sast._CLASS_LABELS` gains the label. **No-echo invariant:** the snippet MASKS the value (first/last few chars only) so the raw credential is never written into the finding message, the check-run, or the logs; the judge discriminates real-value-vs-placeholder from the full-file context (#336) it already receives, so a known docs example (`AKIA...EXAMPLE`) is SUPPRESSED with a reason while a real leaked key publishes advisory-by-default. `dispatch_code_review` concatenates SAST + SCA + secret candidates into the ONE `judge_candidates` pass; best-effort + additive throughout (a detector failure returns `()` and never breaks the core review). Diff-scoped, so a secret already on an unchanged line is not re-flagged. |
| **IaC misconfiguration scanning (#447, ADR-0007 Track 1)** | Third Track-1 slice (the IaC half of slice 2): flag insecure infrastructure-as-code a PR INTRODUCES on added lines, as another candidate SOURCE feeding the same exploitability-judge + publish pipeline. `iac_scan.py` (mirrored): `scan_iac(hunks) -> tuple[Candidate, ...]` is pure, diff-scoped, and FILE-TYPE-AWARE - Terraform patterns (`public-read` ACL, `publicly_accessible`, encryption/deletion-protection off) fire only on `*.tf`/`*.tfvars`, Kubernetes/compose patterns (`privileged: true`, `hostNetwork/hostPID/hostIPC`, `runAsNonRoot: false`, `readOnlyRootFilesystem: false`, TLS-verify off) only on YAML, Dockerfile patterns (`USER root`, `:latest` base, `ADD <url>`) only on Dockerfiles; cross-cutting rules (open `0.0.0.0/0` + `::/0`, `chmod 777`, pipe-to-shell) on any IaC file. Cost-bounded like secret_scan (cap 100 + per-line + total-byte budgets), content-deduped per `(file, kind, match)`. Emits `Candidate(vuln_class="iac-misconfig")`; the judge decides exploitability (an intentional public ALB vs an exposed admin port) so kept findings publish advisory-by-default. |
| **Lambda exit complete (#354 / infra#1170+#1172)** | webhook flipped + Lambda/poller/ECR retired 2026-06-12 (live-proven: real redelivered delivery -> pod -> Elder check-run; GitHub deliveries went 500->200 because the route annotation bug shipped fixed on k8s). api flipped via the same SSM upstream override; the k8s poller CronJob unsuspended as the ONLY poller. The CF Worker remains the auth boundary for BOTH hosts; both grug.lol DNS records are hand-owned pending the infra zone migration (the stack token lost zone DNS - infra#239). grug-main DDB has no readers post-api-retirement; table retirement = operator decision. |
| **`parse_diff`** | Pure function in `personas/code_reviewer/diff_parser.py` (mirrored — ADR-0001) that takes a unified-diff string and returns `tuple[DiffHunk, ...]`. Hand-rolled (no `unidiff` dep) — the subset we need is small and Lambda cold-start cost of the dep wouldn't pay back. Handles multi-file diffs, skips binary blocks, picks the new-side path on renames. Spec 0015 §Parse contract attests purity. |
| **`DiffHunk`** | Frozen dataclass — fields `file_path: str`, `new_start: int`, `new_lines: frozenset[int]`, `body: str`. `new_lines` is the set of new-file line numbers that were added or are context-with-a-removed-neighbor; the Elder anti-hallucination filter rejects LLM findings whose `(file, line)` is not in this set. `body` retains the raw @@-prefixed text for feeding back to the LLM as review context. |
| **Full-file context (#336)** | The Elder + the judge see the WHOLE current content of each changed file (fetched at head SHA via `GET /contents/{path}?ref=` with the `.raw` Accept header), not just the diff hunk. `dispatch._fetch_file_contents` collects them (capped at `_MAX_CONTEXT_FILES=20`, best-effort — a per-file failure degrades that file to diff-only, never blocks the review); `llm_client._build_messages` / `_build_judge_messages` prepend the 1-based numbered file content (once per path) before the diff, skipping files over `_MAX_FILE_CONTEXT_LINES=800` to bound token cost. The Elder flags ONLY diff additions but READS the whole file, so a mitigation outside the changed lines — a later `finally`/`if: always()` cleanup, a guard above — is visible. Fixes the false-positive class where the cure lives outside the hunk (infra PR #1149: `resource-leak` flagged with an `if: always()` rm 50 lines down; `secret-in-log` flagged for a PATH-not-value env export). Paired prompt wisdom in `code_review_prompt._MITIGATION_SCAN` (scan-before-flag + path-isn't-secret). Backward-compatible: empty `file_contents` ⇒ byte-identical pre-#336 diff-only prompt. |
| **`evaluate_diff`** | Pure function in `personas/code_reviewer/persona.py` (mirrored) that consumes `tuple[DiffHunk, ...]` + `LlmReviewResponse` (from `llm_client.py`) and produces a `CodeReviewEvaluation`. Drops findings outside the diff (anti-hallucination); maps the wire-format `llm_client.Finding` (`path`, `rule`) to the persona-level `Finding` (`file`, `rule_name`, plus `suggestion`). Spec 0015 §Evaluate contract attests purity. |
| **`Finding` (persona-level)** | Frozen dataclass distinct from `llm_client.Finding`. Fields: `file`, `line`, `severity: Severity`, `rule_name`, `message`, `suggestion`. Posted as GitHub inline review comments by the publisher slice. |
| **`Severity` / `review_types.py`** | `Severity = Literal["low","medium","high","critical"]` + `SEVERITIES = frozenset(get_args(Severity))` live in the shared leaf `review_types.py` (mirrored, imports nothing internal) — the SINGLE source (#250). `llm_client`, `persona.py`, and `code_review_prompt.py` all import it (down to the leaf, no cycle), replacing the three independent copies that could silently drift. |
| **`CodeReviewEvaluation`** | Aggregate verdict from one Elder pass. Frozen dataclass — fields `findings: tuple[Finding, ...]`, `passed: bool`, `conclusion: CheckConclusion`. `passed = no high+critical findings`; medium+low are advisory. When the LLM didn't produce content (`kind in {no_diff, all_failed, parse_failed}`), `passed=True` + `conclusion=neutral` — Elder is advisory-first, infra flakiness must not block PRs. Composes 1:1 into a `CheckRunResult` (spec 0001). |
| **`post_review`** | Module-level function in `github_reviews_client.py` (mirrored) that POSTs a `ReviewResult` to `POST /repos/{owner}/{repo}/pulls/{pull_number}/reviews`. Same pattern as `post_check_run`: no retry/no swallow — `with_install_token_retry` is the caller's responsibility. Spec 0016 §Schema + §Retry contracts. |
| **`ReviewResult`** | Frozen dataclass — fields `commit_id: str`, `event: Literal["COMMENT", "REQUEST_CHANGES"]`, `body: str`, `comments: tuple[InlineComment, ...]`. `event=COMMENT` is the advisory mode; `event=REQUEST_CHANGES` blocks merge. APPROVE + PENDING are intentionally not modeled (an LLM-backed reviewer should not auto-approve, and draft reviews don't publish). `commit_id` pins the review to a specific PR head so it cannot race against new pushes. |
| **`InlineComment`** | Frozen dataclass — fields `path: str`, `line: int` (NEW-side line, matches `DiffHunk.new_lines`), `body: str`. Constructor asserts `line >= 1` because GitHub's PR Reviews API 422s on `line=0`. |
| **`ReviewEvent`** | Literal type — `"COMMENT" \| "REQUEST_CHANGES"`. Restricted subset of GitHub's full enum (APPROVE/COMMENT/REQUEST_CHANGES/PENDING) — see `ReviewResult` for why. |
| **Advisory/blocking gate (`_publish_shape`)** | Single source of truth for the dispatch mode → publish mapping, in `personas/code_reviewer/dispatch.py` (mirrored). `mode = "blocking" if RepoConfig.code_reviewer_blocking else "advisory"`. `_publish_shape(evaluation, mode)` returns ONE aligned `(check_conclusion, review_event)` pair so the check-run and the inline review can't drift: advisory mode OR any `degraded_reason` → `("neutral", "COMMENT")` (Elder cannot block on infra flakiness); blocking + `conclusion=="failure"` → `("failure", "REQUEST_CHANGES")`; else `(conclusion, "COMMENT")`. Spec 0017 §Gate contract attests this. |
| **Dual-publish independence** | `dispatch_code_review` publishes the check-run (`post_check_run`) and the PR review (`post_review`) on INDEPENDENT surfaces — each in its own try/except; a 5xx on one records a `*_publish_failed` flag and falls through to attempt the other (no early return). `_resolve_result` consults BOTH flags. Every upstream failure (diff fetch, parse, LLM outage, publish 5xx) degrades to advisory-neutral via `_publish_degraded` — `dispatch_code_review` never raises a wire-level exception, so the persona can't 500 the webhook handler. Spec 0017 §Publish-independence contract attests this. |
| **Finding dedup** | On a `synchronize`/`reopened` push the Elder re-reviews the whole diff; `personas/code_reviewer/dedup.py` (mirrored) stops it re-posting inline comments on unchanged lines. Each inline comment carries a hidden `<!-- grug-rule:NAME -->` marker; `prior_keys_from_comments` parses the PR's existing Grug comments into `(file, line, rule)` keys; `dedup_findings` drops new findings whose key matches. Moved/changed lines get a different key → posted as new. Dedup affects only the inline REVIEW; the check-run summary/conclusion still reflect ALL current findings. Fetch failure → empty key set → post everything (a duplicate beats skipping the review). |
| **Prompt library** | The Elder review system prompt, in `code_review_prompt.py` (mirrored, a sibling of `llm_client.py` — its consumer — so no import cycle and no persona-up import). `ReviewRule` frozen dataclass (name / bug_class / description / bad+good example / default severity); `RULES` is ≥15 bug-class rules seeded from the /audit skill stages; `build_system_prompt(variant: PromptVariant = "v1")` renders them deterministically + appends the `Finding` JSON output contract. **Prompt A/B (#191):** the prompt splits into a shared head/tail + a variant-specific confidence clause — `v1` (precision-biased: prefer a false negative, omit when unsure; byte-identical to the shipped prompt) and `v2` (recall-biased: surface at MEDIUM confidence). `llm_client._SYSTEM_PROMPTS = {"v1": …, "v2": …}` built at import. `select_prompt_variant(installation_id)` maps an install to an arm per the experiment mode (`off`→all-v1 / `split`→per-install / `all_v2`); the split bucket uses `(id // 2) % 2`, **orthogonal** to the backend split's `id % 2`, so the two arms can't confound. The mode is a fallback-safe `/grug/elder-prompt-experiment` SSM String (`secrets_loader.get_prompt_experiment_mode` → `off` on any error). The chosen `variant_id` is annotated on the review's DD LLM Obs span so eval results slice by arm. |
| **LLM-input secret redaction (#438)** | The default review/judge backends are SaaS (Poolside / OpenRouter), and a PR diff can carry a committed credential, so `_build_messages` masks secret-shaped values (`_redact_secrets`: AWS / GitHub / Slack / OpenAI / OpenRouter tokens, PEM keys, `KEY=VALUE` env lines) in the user content BEFORE it reaches `_call_backend` - the MAIN review never transmits a raw secret off-boundary. The Elder reviews code structure, not the literal value, so masking costs no review quality, and a diff with no secret-shaped tokens is byte-identical (the redactor substitutes only on a pattern match - a secret-shaped but benign token is also masked, an accepted false-positive per `_redact_secrets`). Previously `_redact_secrets` guarded only the DD LLM Obs span payload, not the backend call. SCOPE: this covers the main review only; the exposed-secret JUDGE still needs the value to tell a real key from a docs example, so redacting `_build_judge_messages` + moving that judgment in-house to the Cave is tracked separately (#439). |
| **LLM-as-a-judge** | Quality-feedback loop. After a review publishes, `personas/code_reviewer/judge.run_judge` grades each surviving finding via a SECOND LLM call (`llm_client.judge_findings`, own `elder_judge` DD span) and submits a per-finding `is_real_bug` categorical evaluation to DD LLM Obs (`llm_client.submit_finding_evaluation`), attached to the original review span. Best-effort: runs post-publish, never raises, never alters the review outcome. Gated on findings-present + review-span-present. Builds ground-truth for prompt optimization. |
| **`FindingJudgement`** | Frozen dataclass — fields `finding_index: int`, `is_real_bug: bool`, `reasoning: str`. Returned by `judge_findings`; `finding_index` ties back to the evaluation's finding list. Out-of-range / missing indices are dropped (anti-hallucination, same discipline as `evaluate_diff`). |
| **`review_span_context`** | Optional exported DD LLM Obs span carried on `LlmReviewResponse`. The span of the successful review call — the judge attaches `is_real_bug` evals to THIS span so the eval shows on the trace whose output produced the finding. `None` when the review degraded or ddtrace is absent (judge then skips). |
| **Reaction-poll loop** | Second quality-feedback loop (#245). A developer's thumbs-up/down on a Grug inline comment is the HUMAN ground-truth that calibrates the LLM judge. GitHub does NOT webhook comment reactions, so the **`grug-poller` Kubernetes CronJob** (every ~15 min, `k8s/poller-cronjob.yaml`, entry `poller_handler.handler`) reads them via the reactions API per allowlisted install and submits `human_verdict` evals to DD LLM Obs. The poller also best-effort replays missed webhook deliveries each run (#407, `replay_since`). |
| **`CommentRecord`** | Store row persisting a Grug inline-comment for later reaction polling. Populated by **capture-on-publish** (#247a): after `post_review` succeeds, `dispatch` fetches the review's per-comment IDs (`github_reviews_client.get_review_comments` — the create-review response omits them), matches each to its `Finding` by (file, line, rule) — the rule recovered from the comment's hidden `grug-rule:` marker, so two rules on one line don't collapse — and `put_comment_record`s it keyed to the review span + the judge's `eval_tags`. Best-effort + post-publish (spec 0017 `comment_capture_is_best_effort_post_publish_per_dispatch`): a capture failure never alters the review. `PK=INST#<id>`, `SK=CRCOMMENT#<comment_id>`; carries repo, pr_number, `review_span_context`, finding_tags, and `last_verdict` (dedup baseline). `install_store.put_comment_record` / `list_comment_records` / `update_comment_record_reaction`. The poller only submits a `human_verdict` when the current reaction classification differs from `last_verdict` — a stale 👎 doesn't re-submit every cycle, but a 👎→👍 flip does. |

## Activity feed — check verdicts (PRD #301, Slice S1)

| Term | Definition |
|---|---|
| **`Check verdict` / `CheckVerdictRecord`** | The recorded outcome of one **Persona**'s check-run on one PR — the atom of the Activity feed ("What Grug did"). Persisted by `install_store.put_check_verdict` / read by `list_check_verdicts` (mirrored, ADR-0001). Single-table: `PK=INST#<install_id>`, `SK=ACT#<head_sha>#<persona>`. **Idempotent per `(persona, head_sha)`**: re-reviewing the same commit upserts (heals the row), a new commit appends. Stores RAW facts (`persona` caveman key, `repo`, `pr_number`, `head_sha`, `conclusion`, `summary`, `findings_count`, `blocking`, `created_at`, sparse `degraded_reason`) + a denormalized `verdict` for cheap filtering — the raw facts are canonical (ADR-0003). 90-day `ttl` bounds the `ACT#` partition; `list_check_verdicts` sorts by `created_at` desc in SQL (the SK encodes `head_sha`, not time). |
| **`Verdict` / `verdict()`** | The at-a-glance badge — `block` / `warn` / `pass` / `errored` — derived from a Check verdict's raw facts by the SINGLE pure function `review_types.verdict(conclusion, findings_count, degraded_reason)` (ADR-0003). Precedence: `degraded_reason` set → `errored` (never `pass` — "no lies"); `conclusion=="failure"` → `block`; conclusion NOT in {`success`,`neutral`} (cancelled/timed_out/action_required/skipped/stale — Grug never concluded) → `errored`; `findings_count != 0` (any non-zero, incl. a defensive negative) → `warn`; else `pass`. Both the write path and the future `/activity` read path call this ONE function; the frontend renders it verbatim (single source of truth). |
| **Persona name mapping (`chief`/`elder`)** | `review_types.persona_for_key` / `key_for_persona` map the legacy code keys (`tpm`/`code_reviewer`) ↔ the canonical caveman names (`chief`/`elder`, ADR-0002). The ONE boundary the map lives at; `CheckVerdictRecord.persona` stores the caveman name. |
| **`GET /activity` (read endpoint, S2)** | `installations.list_activity` (api service only — the SPA's read path, not mirrored): `GET /api/v1/installations/{install_id}/activity`. Session-authed via `_ensure_can_access` (404 unknown install, 403 not-yours). Returns `{activity: [...]}` newest-first. The `verdict` badge is **re-derived server-side** from each row's raw facts via `review_types.verdict` (ADR-0003 — NOT the stored denorm, so a mapping change heals history on read); the SPA renders it verbatim, never re-derives. `?verdict=` filters to one badge (fetches a wider window so the cap applies to MATCHING rows); `limit` caps (default 50). The web `ActivityPanel` calls it via the `useActivity` hook, replacing the honest empty state with real rows + the ERRORED badge + ALL/BLOCK/WARN/PASS/ERRORED filter chips. |
| **`record_check_verdict` (best-effort writer)** | `activity_log.record_check_verdict` (mirrored) is the ONE place both dispatchers write a Check verdict: map the persona key, derive the `verdict`, stamp `created_at`, upsert. **Never raises** — a recording failure logs `check_verdict_record_failed` and is swallowed so it can't break the check-run the dispatcher just published (same discipline as capture-on-publish + the #272 offload). Chief writes after `publish_tpm_evaluation` (`findings_count` = failed BLOCKING DoR checks); Elder writes after dual-publish (`findings_count` = surviving findings, `conclusion` = the PUBLISHED conclusion so advisory high/critical reads `warn` not `block`) and on the fetch/parse-degraded path (→ `errored`). |

## Elder cave fallback — SQS airlock (ADR-0005, spec 0018)

When **both** cloud LLM backends fail (`review_diff` → `kind=all_failed`), Elder's owned fallback is **"the Cave"** — the operator's self-hosted, OpenAI-compatible LLM, reachable only over the operator's private network (never publicly exposed). It is reached over an **SQS airlock** so the Cave is never exposed and GitHub credentials never leave AWS. Grug and the Cave never connect; they only ever touch the queues. **Part A (#310) is the AWS side, small diffs inline**; the `grug-cave-connector` worker (#316) and the live cutover (#313) are separate slices. The mechanics of the private connection live in the operator's private infra repo.

| Term | Definition |
|---|---|
| **`grug-cave-jobs` / `grug-cave-results`** | The airlock — two SQS **FIFO** queues (Pulumi). `jobs`: webhook pod -> connector. `results`: connector -> webhook (long-polled by `consumer.py`). FIFO gives free content-dedup + per-install ordering (`MessageGroupId=install_id`); **long polling** (`receive_wait_time_seconds=20`, free-tier guard) + **DLQ + redrive** (`maxReceiveCount` 5 jobs / 3 results → `grug-cave-{jobs,results}-dlq`) + DD monitors (DLQ depth, queue age, fallback-fired rate) are the resilience hardening (#312). The producer dedups **head-scoped** (a new push = a distinct fallback — unlike the rerun's head-less dedup). SQS is greenfield in grug here; ADR-0004's rerun queue (#305) reuses this scaffolding (extraction deferred, rule-of-three). |
| **`FallbackJob` / `FallbackResult`** | The cross-repo message contract, now the **shared `spark_cave` envelope** (vendored from infra-public at `services/webhook/spark_cave/`, SHA-pinned via `VENDOR.md`; #454/infra#1610 — do not edit in grug, fix upstream + re-vendor). `FallbackJob{schema_version, persona, principal_id, request_id, payload: inline-or-S3 ref}` with `dedup_key = persona:principal:request`; `FallbackResult{..., ok, result, error}`. grug maps persona=`elder`, principal=install id, request=`repo:pr:head_sha`, and rides its rich fields (`payload_version=2`, diff_ref) inside `payload`. The envelope is persona-generic — a second lane (e.g. an in-house judge, #439) is a schema-free addition. |
| **`DiffRef` codec** (`pack_diff`/`unpack_diff`, #311) | The deep module behind the airlock's diff delivery. A `DiffRef` is a JSON dict that's EITHER `{"kind":"inline","hunks":[…]}` (small PR) OR `{"kind":"s3","bucket":…,"key":…}` (large PR). `pack_diff(hunks)` inlines under `_MAX_INLINE_DIFF_BYTES` (250 KB), else spills the hunks to the `grug-cave-diffs` S3 bucket (`diffs/<install>/<head>.json`, 7-day lifecycle) and returns the pointer; `None` when a big diff can't spill (no bucket / put failed → enqueue no-ops, best-effort). `unpack_diff(diff_ref)` reconstructs the hunks (inline payload or S3 fetch) — the webhook packs, the connector (#316) unpacks, so a big diff is read from S3, **never re-fetched from GitHub** (the creds boundary). IAM: webhook `s3:PutObject` + connector `s3:GetObject`, scoped to `diffs/*`. |
| **`enqueue_fallback` (producer)** | `cave_fallback.enqueue_fallback` (webhook-only). Called from `dispatch_code_review` **only** when `llm_response.kind=="all_failed"` AND `secrets_loader.get_fallback_enabled()` is true (SSM `/grug/elder-fallback-enabled`, fallback-safe → `false`). `SendMessage` to `grug-cave-jobs` with `MessageDeduplicationId` from `(install,repo,pr,persona)` + `MessageGroupId=install_id`. **Best-effort / never raises** — a SendMessage failure logs `elder_fallback_enqueue_failed` and is swallowed (the degraded `errored` verdict was already published; the fallback re-triggers on the next push). The verdict stays `errored` until healed (the "no lies" rule). |
| **`handle_fallback_result` (consumer)** | `cave_fallback.handle_fallback_result` (webhook-only). `consumer.py` long-polls `grug-cave-results.fifo` and feeds each message here wrapped in the `Records` batch shape. Parses a `FallbackResult` (shared `spark_cave.schema`), `post_check_run` with the connector findings, then `record_check_verdict` — which upserts idempotently per `(persona, head_sha)`, **healing `errored` -> `reviewed`** for the current head (a moved-on PR appends a fresh row). **Never raises back to the consumer** (a raise would retry-storm the queue); a malformed/degraded result is logged and dropped. |

## Re-run errored rows — SQS backfill (#305, ADR-0004)

grug's backfill for a dropped/`errored` review (the 2026-06 Elder outage left reviews unre-runnable — webhook redelivery is a `claim_delivery` no-op). Reuses the cave SQS scaffolding + deploy-role grants.

| Term | Definition |
|---|---|
| **`grug-rerun-jobs` (+ DLQ)** | SQS **FIFO** queue + DLQ (redrive `maxReceiveCount=3`, visibility 420s). api → enqueue (`rerun.enqueue_rerun`), webhook → consume (event-source mapping). Content-dedup on `(install, repo, pr, persona)` — a double-click drops; `MessageGroupId=install_id` rate-controls a batch backfill. DLQ-depth DD monitor. Keyed by **`repo`** ("owner/name", what the Activity row carries) — NOT a repo_id, which the verdict record doesn't store. |
| **`POST /installations/{id}/rerun`** (api) | `{repo, pr_number, persona}` → **202** + enqueue. Auth via `_ensure_can_access` (owns install / admin). 503 if the queue isn't configured. The api pod hands off to the durable queue; the webhook consumer runs the review on its own budget. `RerunRequest` pattern-constrains `repo` + `persona` (422 junk). |
| **`rerun.handle_rerun_jobs`** (webhook) | `consumer.py` long-polls `grug-rerun-jobs.fifo` and routes each batch here. Fetches the PR's CURRENT head (+ `base.repo.id` for RepoConfig) and re-runs the named persona via the unchanged `dispatch_code_review` — which upserts the verdict (heal-in-place on an unchanged head, append on a moved-on PR). Unlike the cave handler, a transient infra failure **RAISES** so the ESM retries (visibility) → DLQ; a repeat LLM outage degrades (published `errored`) and the job is "done". Code-reviewer persona only today; TPM rerun is a follow-up (the static check doesn't error from outages). |
| **`↻` re-run button** (web) | On **`errored`** Activity rows only. Click → `RE-RUNNING…` + `useRerun` POST; the feed refreshes ~4s later to pick up the healed verdict. |
| **`POST /installations/{id}/rerun-all`** + batch button (#306) | The outage-recovery move ("Elder failed on 12 PRs → one click"). Re-derives the verdict for every row server-side and fans each DISTINCT `(repo, pr, persona)` `errored` row into `grug-rerun-jobs` (de-duped across heads); returns `{queued: N}`. The per-install FIFO `MessageGroupId` paces the batch (no concurrent LLM burst) + content-dedup drops any row already in-flight from a single re-run. Web: a `↻ Re-run all errored (N)` button marks every errored row `RE-RUNNING` and each flips to its verdict as the queue drains. |

## Enforcement concepts

| Term | Definition |
|---|---|
| **GitHub Ruleset** | A GitHub Repository Ruleset that requires specific status checks to pass before merging. Grug creates rulesets to enforce its DoR check on the default branch. Managed via the Repository Rulesets API (`POST/GET/DELETE /repos/{owner}/{repo}/rulesets`). See [`services/{api,webhook}/github_rulesets_client.py`](services/api/github_rulesets_client.py) (mirrored — see ADR-0001). |
| **Grug-managed ruleset** | A ruleset whose `name` starts with the prefix `Grug —`. This prefix is the ownership marker: `detect_enforcement()` uses it to distinguish rulesets Grug created from externally-managed ones. |
| **EnforcementState** | Literal type — `"grug_managed" \| "external" \| "none"`. Returned by `detect_enforcement()`. `grug_managed`: at least one `Grug —`-prefixed ruleset exists with `required_status_checks` matching the check name. `external`: no grug-managed ruleset, but the check is enforced via a non-Grug ruleset or legacy branch protection. `none`: check is not enforced anywhere. |
| **`detect_enforcement()`** | Module-level function in `github_rulesets_client.py` that determines the enforcement state for a given status check. Queries both the Rulesets API and the legacy Branch Protection API (`GET /repos/{owner}/{repo}/branches/{branch}/protection/required_status_checks`) to cover repos that haven't migrated to rulesets. **Resilient GETs:** both reads go through `_get_with_retry` — bounded jittered-exponential-backoff retries on GitHub's secondary rate limit (429, rate-limit-403, 5xx) + transport errors, honoring a (capped) `Retry-After`, with `rulesets_get_retry`/`_exhausted` logs. The dashboard fans one `/enforcement` request out per repo in parallel, which tripped GitHub's secondary limit → a false "not enforced"; the retry budget is small (fits the api request budget) and a SUSTAINED limit is handled by the endpoint fallback, not a long wait. |
| **Enforcement detection fallback** | `get_enforcement` (api) wraps live detection: on a rate-limit/transport error that survives the retries, it degrades to the **last-known stored state** rather than 500ing or rendering a false "none" — `grug_managed` if a `enforcement_ruleset_id` is stored, else `unknown` (a distinct `EnforcementState` the badge shows as muted "status unknown", never "⚠ not enforced"). Response carries `degraded: true`. A legitimate `404` (repo not found) is NOT caught (only `httpx` errors degrade). The dashboard hook (`useEnforcement`) additionally retries with **jittered** backoff so its parallel retries don't re-burst. Logged as `enforcement_detect_fallback`. |
| **Legacy branch protection** | Pre-rulesets mechanism for requiring status checks. Still active on many repos. `detect_enforcement()` checks this as a fallback when no ruleset-based enforcement is found. |
| **`enforcement.py`** | Mirrored module containing enforcement lifecycle functions. `ensure_enforcement()` detects current state and creates a Grug-managed ruleset if none exists; `remove_enforcement()` deletes it. Both persist the `enforcement_ruleset_id` on the `RepoConfig` store row. See [`services/{api,webhook}/enforcement.py`](services/api/enforcement.py). |
| **`ensure_enforcement()`** | Idempotent function: detect → skip if `grug_managed` or `external` → create ruleset → store `enforcement_ruleset_id`. Called on installation created (webhook) and persona enable (API toggle). |
| **`remove_enforcement()`** | Deletes the Grug-managed ruleset by ID (read from `RepoConfig`) and clears the stored `enforcement_ruleset_id`. Called on persona disable (API toggle). |
| **`enforcement_ruleset_id`** | Optional integer field on the `RepoConfig` store row. Stores the GitHub ruleset ID of the Grug-managed enforcement ruleset for quick lookup during delete. `None` means no Grug-managed ruleset exists. |
| **`force_disable_enforcement`** | Optional boolean field on the `RepoConfig` store row, default `False`. When `True`, the self-healing loop skips re-creation of a deleted Grug-managed ruleset. Escape hatch for users who intentionally remove enforcement without disabling the TPM persona. |
| **`heal_enforcement()`** | Module-level function in `enforcement.py`. Called from dispatcher when a `repository_ruleset` deleted event fires for a Grug-managed ruleset. Clears the stale `enforcement_ruleset_id`, delegates to `ensure_enforcement()` for idempotent re-creation, and emits an `enforcement_healed` structured log with old + new ruleset IDs. Skipped when `force_disable_enforcement` is `True` or TPM persona is disabled. |
| **Self-healing** | Reconciliation loop: when a Grug-managed ruleset is externally deleted, the webhook re-creates it if the repo still wants enforcement. Triggered by `repository_ruleset` webhook event with `action=deleted`. The `force_disable_enforcement` flag on `RepoConfig` is the opt-out. |
| **Enforcement migration** | One-time backfill script (`scripts/migrate_enforcement.py`) that scans all installations for TPM-enabled repos and creates Grug-managed rulesets where none exist. Handles the grug repo's legacy branch protection → ruleset migration. Supports `--dry-run`. Idempotent. |
| **`grug.enforcement.state`** | DogStatsD gauge metric emitted by `emit_enforcement_metric()` in `observability.py` on every enforcement state change. Value: 1.0 (grug_managed), 0.5 (external), 0.0 (none). Tags: `repo`, `persona`, `enforcement_type`. Used by the enforcement gap monitor. |
| **Enforcement gap monitor** | DD monitor `grug-enforcement-gap` that alerts when any repo has `enforcement_type:none` for >1 hour. Routes to the DD monitoring Discord webhook. |

Until Roles Anywhere lands (#388/#389), the `grug-key-rotator` CronJob bounds the exposure window by rotating that key every 12h. Logic in `services/webhook/key_rotator.py` (`rotate()` - injectable `iam`+`k8s`, fully unit-tested). Safety invariant: **a pod never runs on a deleted key** - ordering is create-new -> patch `grug-secrets` -> rollout-restart api/webhook/consumer and WAIT -> only then delete-old. Fail-safe-open: any failure before the new key is live keeps the old key valid (the `key_rotation_failed` log + DD monitor surface it; a dangling new key may need manual cleanup since AWS caps a user at 2 keys). The rotator authenticates with its OWN credential (`grug-rotator-secret`, IAM user `grug-k8s-rotator` scoped to `*AccessKey` on `grug-k8s-pod` only - cannot escalate), NEVER `grug-secrets`. It is the only namespace workload with k8s API access (a namespace-scoped Role over exactly the one Secret + three Deployments). Removed wholesale when Roles Anywhere ships (#389). |

## Operational concepts

| Term | Definition |
|---|---|
| **Dependency-aware readiness (#404)** | `/readyz` means "this pod can do its job", not just "process up". The `readiness` module (mirrored) probes the CRITICAL dependencies - SSM+KMS (a SecureString read that exercises AWS auth + KMS decrypt in one call, so a deleted/invalid pod key is caught) and Postgres (`SELECT 1`) - TTL-cached (~5s) and **fail-closed** (any error -> not ready, never falsely ready). The api + webhook `/readyz` handlers return 503 when any dependency is unreachable. Effect: a dependency-broken pod fails readiness, so k8s stops routing to it AND a rollout of broken pods never completes - turning a silent cluster-wide outage (the 2026-06-14 deleted-key incident) into a self-limiting deploy. This "last-good pod keeps serving" property requires the deploy strategy be `RollingUpdate` with `maxUnavailable: 0` (the new pod must pass readiness before the serving pod is removed); api + webhook were switched from `Recreate` to exactly that (#404). The probe is fail-FAST (tight SSM timeouts/retries + a bounded, pool-independent Postgres connection) so a slow dependency or busy pool can't blow the kubelet probe timeout and drop the replica. `/livez` stays a cheap process-up check so a transient dependency blip restarts nothing. Probe `timeoutSeconds` is raised (cache-miss does real I/O). The queue consumer (no HTTP) gets the equivalent via a fail-fast startup self-check (#405, below). |
| **Consumer fail-fast startup self-check (#405)** | The `grug-consumer` process has no HTTP server, so the kubelet has no `/readyz` to gate it on - a pod with a broken AWS key would start its poll threads, hit the receive-error backoff forever, and sit idle "Running" while its queues silently back up. `consumer._startup_check()` runs FIRST in `main()` (before any poll thread spawns) and calls the SAME `readiness.check_readiness()` the `/readyz` handlers use - one definition of "is AWS reachable", no second copy. If any critical dependency is unreachable it logs `consumer_startup_check_failed` and `raise SystemExit(1)`, surfacing as a visible CrashLoopBackOff the kubelet paces. This complements the pre-existing thread watchdog (a poll thread that dies mid-run also exits non-zero); together they cover both "broken before the first poll" and "broke during a poll". |
| **k8s-native alerting (#406)** | The Lambda->k8s migration left the core monitors querying retired `aws.lambda.*` metrics (`errors`/`invocations`/`enhanced.init_duration`), so `[grug-webhook] 5xx`, `[grug-api] 5xx` and `[grug] cold-start p99` sat in permanent `No Data` with `notify_no_data=False` - structurally unable to fire (the 2026-06-14 outage paged nobody). Those three are RETIRED and replaced with signals built on Kubernetes State Metrics, which DO flow for the grug namespace: workload-not-ready (`kubernetes_state.deployment.replicas_ready{kube_namespace:grug} by {kube_deployment} < 1` - covers webhook/api/consumer being down, the availability equivalent of the old 5xx alert), CrashLoopBackOff (`kubernetes_state.container.status_report.count.waiting{kube_namespace:grug,reason:crashloopbackoff} by {pod_name} > 0` - covers grug-consumer, the workload that crash-looped unmonitored), and a restart-spike signal. KSM queries scope by `kube_namespace:grug` (the reliably-present tag), NOT the stack `env` tag (which doesn't propagate onto KSM series - relying on it would reintroduce a `No Data` trap). Folds in #379's poller half: a poller CronJob-health monitor (`kubernetes_state.cronjob.duration_since_last_successful{kube_cronjob:grug-poller}`). #379's consumer queue-age half is DEFERRED: `aws.sqs.*` is not collected by the DD AWS integration in this org, so a queue-age monitor would be permanent `No Data` - the exact trap this slice retires. The consumer's crash/down outage mode is covered by the crashloop + workload-not-ready monitors; a true queue-age signal needs the SQS integration namespace enabled first. Continuous-KSM monitors (workload-not-ready, poller) set `notify_no_data=True` (No Data == the telemetry pipeline broke = page); conditional ones (crashloop, restart-spike) set `notify_no_data=False` (No Data == healthy). Every grug monitor routes to the single `@webhook-grug-discord-monitoring` Discord webhook integration. |
| **Consumer APM writer init (#406)** | grug-consumer emits ZERO APM spans while grug-webhook (same image, same node-local agent) traces fine - and a peer service on the same OCI node traces fine - so it is NOT a node/agent problem (the NetworkPolicy already permits APM egress TCP 8126 and `DD_AGENT_HOST`/`DD_VERSION` are wired). The pre-#405 consumer logged `ddtrace ... failed to start writer service` from a botocore SQS call on a poll thread: the spans that matter are created on worker poll threads, where ddtrace's lazy writer-service start can fail and silently drop spans. Mitigation (`consumer._warm_trace_writer`): open one ddtrace span on the MAIN thread before the poll threads spawn, initializing the writer in the main-thread context. Efficacy is confirmed by checking grug-consumer spans appear in Datadog AFTER deploy; if they still don't, the root cause is elsewhere and tracked separately (not assumed fixed). Fail-safe: telemetry never breaks consumer startup. **#412 follow-up:** the warmup fixed the main-thread startup span but the worker-thread per-poll SQS spans still did not reach Datadog over hours, while a local repro under the same ddtrace (3.19.7) could NOT reproduce a worker-thread bug (worker spans flush fine there) - so the prod cause is environmental, not span-creation. Fix: `consumer._flush_traces()` calls `tracer.flush()` from the MAIN thread (the path proven to deliver) on each watchdog tick, draining the shared buffer so worker-thread spans are delivered regardless of what stalls the long-running periodic writer. |
| **Missed-delivery replay (#407)** | The DoR/TPM check runs INLINE on the webhook, so a GitHub delivery that arrives while grug is down is lost - GitHub retries a few times then gives up, leaving a required check stuck until a human re-triggers (infra #1254). Recovery uses the GitHub App webhook-deliveries API (App-JWT, no installation token): `GET /app/hook/deliveries` lists recent deliveries (cursor-paginated, newest first; a delivery is SUCCESS when `status_code` is 200-399, FAILED at 400-599 or a connection error). The deep module `delivery_replay` groups deliveries by `guid` (VERIFIED 2026-06-15 against the live App: redelivering a delivery produces a new row with `redelivery=true` sharing the original's guid) and `POST /app/hook/deliveries/{id}/attempts` re-sends only guids where NO delivery succeeded - so a guid whose redelivery later succeeds is skipped on the next pass (idempotent WITHOUT persistence), and because the redelivered webhook carries that guid as `X-GitHub-Delivery`, the consumer's existing `claim_delivery(delivery_id)` dedupes a second processing. (NB: `post_check_run` POSTs the Checks *create* endpoint and is NOT idempotent on (name, head_sha) - duplicate posts create duplicate check runs - so correctness rests on the guid-skip + claim_delivery, never on re-posting being a no-op.) Auto-recovery: the `grug-poller` CronJob (every 15m) calls `replay_since(window)` best-effort each run, so missed deliveries self-replay with no recovery-detection plumbing. Manual fallback: `services/webhook/replay_deliveries.py --since T` via kubectl exec (runbook-documented). The async SQS layer (grug-rerun-jobs / grug-cave-results) is already resilient - messages persist with a visibility-timeout+DLQ and the consumer auto-drains on recovery (#368/#405) - so #407 adds NOTHING there beyond a documented note. |
| **Cutover** | Migration path from a self-hosted to a hosted setup, or between Pulumi stacks. Runbook at [`docs/CUTOVER.md`](docs/CUTOVER.md). |
| **HITL prerequisite** | A human-in-the-loop step that must precede automated work (e.g. App registration on github.com, SSM secret pre-load). Documented in [`docs/HITL_PREREQUISITES.md`](docs/HITL_PREREQUISITES.md). |
| **Self-host** | Operator deploys Grug against their own AWS + Cloudflare account. Step-by-step in [`docs/SELF_HOST.md`](docs/SELF_HOST.md). AGPL-3.0 network-service compliance applies. |

## Appendix — Slice plan tracking convention (post `/temper-improve`)

SDD slices are planned + executed against the temper specs above. Slice
plans live **in the repo** under
`specs/<NNNN-spec-slug>/plans/<YYYY-MM-DD>-<slice>.md`, not in `/tmp/`
or chat scrollback. Every shipped plan gets an `## Outcome` postscript
capturing the v1→v2 transition (what was proposed, what peer-review
flagged, what shipped). The historical record is the only way future-you
can ask "why does the adapter do X instead of Y?" and get an answer.

**Template:** `specs/SLICE_PLAN_TEMPLATE.md` — copy when starting a new
slice. Bakes the recurring patterns peer-review catches (IOA atomicity,
runtime attester allowlist-not-denylist, cross-user defense at storage
layer, single transactional mutation, clock injection, explicit
deferral checklist, mirror discipline, storage-side scope of mutation)
as required sections so the v1→v2 ceremonial round collapses to
substantive review.

**Workflow:**
1. Copy `specs/SLICE_PLAN_TEMPLATE.md` to
   `specs/<NNNN>/plans/<date>-<slice>.md`.
2. Fill all required sections (or explicit `N/A — <reason>`).
3. Optional but recommended for first slice on a new IOA action: submit
   to `/peer-review` chain. If BLOCK'd, rewrite to v2 in the same file
   with the v1 narrative + v2 deltas preserved as a transition record.
4. Implement against the plan; spec edits are spec-first (TOML → temper
   verify → grounding attester → code).
5. After commit, add `## Outcome` postscript with shipped SHA +
   reviewer-finding pre-emption matrix.

**When a recurring CRIT class becomes mechanically detectable** (e.g.
the over-broad-DELETE pattern from PR #151 could become an AST-walking
attester that flags `delete_item` calls outside test fixtures), the rule
moves OUT of the template INTO a CI gate or grounding attester. Until
then, template + per-slice peer-review are the human-layer enforcement.

**PII guard:** `services/{api,webhook}/tests/test_log_pii_guard.py`
scans for raw secret-bearing field names in log calls (OAuth plaintext
tokens, KMS plaintext keys, App private keys). Does NOT scan for
`github_user_id` / `install_id` — those are intentionally logged with
DD as the authorized sink + support flow needing the raw id. Migrating
identifiers to `observability.fingerprint()` is a deliberate future
call, not a CI blocker today. New log calls referencing the secret-set
field names get red-X'd at PR time unless wrapped with
`observability.fingerprint()` or logging the `_blob` / `_encrypted`
form.
