# DESIGN.md — grug domain design notes (deep-dive companion to CONTEXT.md)

The vocabulary used in `services/`, `infra/`, and `web/`. Terms map to identifiers a contributor will encounter while reading code. New contributors should be able to read this file and use the terms correctly without grepping. Drift between this file and the code is a bug; fix the file in the same PR that renames a concept.

> **Architecture decisions live in [`docs/adr/`](docs/adr/).** This file is the lexicon; the ADRs are the load-bearing choices.

## Product surface

| Term | Definition |
|---|---|
| **Grug** | The bot. A hosted GitHub App at `grug.lol` that gates pull requests against a set of process checks. Open-source AGPL-3.0; self-host path in [`docs/SELF_HOST.md`](docs/SELF_HOST.md). |
| **Grug Boss** | Public-facing name of the GitHub App users install on their repos. Same product as Grug; "Grug Boss" is the GitHub Marketplace listing handle. |
| **Persona** | A bounded behavior surface Grug applies per PR. Two ship: **Chief/TPM** (static Definition-of-Ready check) and **Elder/code_reviewer** (LLM review + the deterministic security suite). Roadmap (epic #464): Guard, Smasher, Omen, Warder, Pulse, and the LLM scope-review half of TPM. Personas are per-repo togglable. |
| **TPM persona** | The process-gate persona (canonical name Chief). Static **Hunt Plan** (legacy: Definition-of-Ready) that blocks merge if PR body is malformed. The companion **Scope review** half is wired as a `poolside_client.py` hook in `evaluate()` per the `__init__.py` docstring but is NOT shipped — see "Roadmap" rows. See [`services/_shared/personas/tpm/`](services/_shared/personas/tpm/). |
| **Elder persona** | The code-reviewer persona. Reads the PR diff and posts inline review comments + a check-run summary. Runs alongside TPM on `pull_request` events (`opened`, `synchronize`, `ready_for_review`, `reopened`), but off the webhook ACK path. The two personas dispatch independently - one failing does not skip the other. Modules: `personas/code_reviewer/diff_parser.py` (unified diff to `DiffHunk`s) + `personas/code_reviewer/persona.py` (`evaluate_diff` rollup) + `personas/code_reviewer/dispatch.py` (orchestration - fetch diff, run LLM, publish via reviews+checks clients). `llm_client.py` defaults to a deep Poolside + OpenRouter ensemble; `GRUG_REVIEW_DEPTH=fast` selects one primary with the other as failure fallback. Advisory by default (`code_reviewer_blocking=False` gives check-run `conclusion=neutral`, review `event=COMMENT`); blocking flips via the dashboard toggle. See specs 0015 + 0016 + [`services/_shared/personas/code_reviewer/`](services/_shared/personas/code_reviewer/). |
| **Pulse (roadmap)** | Scheduled (non-PR-triggered) persona work, named `stuck-PR-pulse` in the future-persona list in `services/webhook/main.py`. Not implemented. Intent: weekly issue-grooming sweep against Grugboard. |

## Process-gate concepts

| Term | Definition |
|---|---|
| **Hunt Plan** | Product name for Chief's PR-body gate (was industry jargon "Definition of Ready"). Enforced as plan checks rolled into `Grug - Chief`. Module path still `dor_checks.py`. |
| **Plan check** | Individual Hunt Plan rule: `why`, `acceptance`, `estimate`, `scope-fence`, `issue-link`. Defined in [`services/_shared/personas/tpm/dor_checks.py`](services/_shared/personas/tpm/dor_checks.py). |

| **TpmEvaluation** | Aggregate result of running all `plan check`s against one PR. Frozen dataclass returned by `evaluate_pull_request(...)` in `personas/tpm/persona.py`. Composes into a `CheckRunResult` for GitHub. |
| **CheckRunResult** | Frozen dataclass that maps directly onto GitHub's Checks API `POST /repos/{owner}/{repo}/check-runs` payload. | **`post_check_run` (publisher)** | Module-level function in `github_checks_client.py` that POSTs a `CheckRunResult` to GitHub. The acceptance-criteria spelling "CheckRunPublisher" is the *concept name* — the actual identifier is a function, not a class. |
| **Scope review (roadmap)** | Advisory LLM pass over PR title + body. Wired as a `poolside_client.py` hook called from `evaluate_pull_request(...)` per the `personas/tpm/__init__.py` docstring, but **`poolside_client.py` does not exist in the repo today** — feature is roadmap-only. Intended behavior: flag title↔body mismatch, AC testability, scope-creep, XL inflation; posted as a comment, never blocking. |

## Elder persona — code review

| Term | Definition |
|---|---|
| **Durable Elder quiet window** | The Elder LLM path runs off the synchronous webhook ACK. `receive_github_webhook` verifies HMAC, runs fast inline personas, then `async_dispatch.enqueue_elder_review` writes a snapshot-scoped `kind=review` message to `grug-rerun-jobs.fifo` and returns `result="queued"`; GitHub is ACKed in under 10 seconds regardless of inference time. FIFO identity covers base SHA, head SHA, title, and body, while message ordering is isolated per PR. Draft events are accepted without enqueueing so `ready_for_review` can enqueue the same content later. The producer stamps `GRUG_ELDER_SETTLE_SECONDS` (90 in production, bounded to 300). The consumer fetches the current PR, leases its snapshot, waits, and fetches it again. A changed snapshot is re-enqueued from the fresh GitHub response and the stale lease is released, so correctness does not depend on a second webhook. Stable work reviews the immutable base/head compare and rechecks the full snapshot plus open/non-draft eligibility before publishing. Model, freshness, and publication failures release the lease and raise for SQS redrive; persistent failures reach the DLQ. A durable enqueue error returns 5xx so GitHub records a failed delivery; GitHub does not retry automatically, and the 15-minute delivery replay poller performs recovery. `GRUG_ELDER_DURABLE_QUEUE` is set only on the webhook deployment; absent/false keeps the local-thread compatibility path. |
| **k8s execution paths (#368)** | `grug-webhook` receives and validates GitHub events; `grug-consumer` owns durable Elder inference. Normal reviews, explicit reruns, and questions share `grug-rerun-jobs.fifo` but use bounded workload groups: review per PR, rerun per PR/persona, and ask per PR. Four rerun poll workers allow those groups to progress concurrently; SQS still serializes each group. Normal reviews enforce the quiet/stale-snapshot gate and renew both their SQS visibility and ownership-token database lease; explicit reruns target the current snapshot immediately. A handler raise leaves the message for visibility-timeout redrive and eventual DLQ. Thread liveness alone is not health: every normal review is registered with an absolute monotonic deadline (720 seconds in production). The watchdog stops renewals and returns all active SQS receipts before releasing their snapshot claims, then hard-exits so kubelet replaces an over-budget or shutdown-stuck worker. This process-level termination is deliberate because Python cannot safely cancel a thread blocked in an upstream HTTP read. Other async personas still use `_AsyncPersonaSpec` and the k8s local worker. Both Elder-capable deployments carry the same review depth/model env so disabling the durable flag does not silently change review intelligence. |
| **Stable-snapshot review / snapshot idempotency (#397)** | A PR update enqueues a message keyed by the canonical `(base SHA, head SHA, title, body)` digest. Elder spends review budget only after one snapshot remains unchanged for the quiet window. Intermediate snapshots in a rapid update sequence are cancelled and the latest eligible snapshot is re-enqueued. The leased review claim uses that digest, so exact redeliveries do not double-post while unchanged-head base or intent edits still receive a new review. Explicit `/rerun` and recovery jobs keep their current-snapshot semantics and bypass the quiet wait. |
| **Tiered Elder review (ADR-0019 / #645)** | `GRUG_REVIEW_DEPTH=tiered` is the production default. Elder always runs the owned Cave **coder** arm with the recall-oriented v2 prompt, full code context, and untrusted PR-intent block. The Cave **reasoner** arm runs only when `decide_deep_escalation` fires (added-line threshold `GRUG_DEEP_DIFF_LINES`, high-risk path markers, explicit `deep-review` in title/body, or deterministic sample `GRUG_DEEP_SAMPLE_RATE`). Dual-arm runs are concurrent and merge with exact duplicate suppression + provenance. `deep` restores always-on dual-arm; `fast` is coder-first with reasoner only if coder fails. SaaS (OpenRouter review-only Opus / Poolside) remains overload fallback only. Judge, Teller, and `/grug ask` keep their shared lower-cost OpenRouter config. |
| **Trusted reaction learning** | Each posted Elder comment persists its redacted finding text, reviewed head, PR author, and every model origin (including origins whose span export failed). For new records, the poller checks each reactor with GitHub's collaborator-permission endpoint and accepts only `write`/`admin` users; external contributors and unrelated users cannot steer feedback. A changed verdict fans `human_verdict` only to the producer spans that actually exist, upserts one stable `grug-elder` ledger row keyed by the comment evidence, and recomputes the repo's bounded practices and few-shot caches. Confirmed feedback becomes positive practice/example evidence. False-positive feedback is excluded from positive examples and becomes bounded AVOID guidance requiring materially new evidence before repetition. Old records without capture fields remain DD-observable but do not auto-train. |
| **SAST benchmark / eval harness (#399)** | A committed, re-runnable corpus + harness that measures Elder's vuln-detection recall + precision per backend - the acceptance gate for the SAST detection slices (#400/#401) and a standing regression guard (ADR-0006). Lives in `services/webhook/sast_benchmark/` (webhook-only tool, NOT in _shared/). **Corpus** (`corpus.py`): one inline diff sample per canonical vuln class (cleartext-secret-log, hardcoded-credential, SQL/command/template-injection, SSRF, path-traversal, unsafe-deser, weak-crypto) each with ground truth `(vuln_class, path, is_true_positive)`, PLUS the known FALSE-POSITIVE shape (the #391 public-config-path log) that must NOT be flagged. **Pure core** (`scoring.py` - CI-safe, no LLM): `score(samples, findings_by_sample) -> BenchmarkReport` (per-class recall = TP samples flagged; precision = TP-flags / all-flags, where any finding on an FP sample is a false positive) + `compare_to_baseline(report, baseline)` (a per-class recall DROP vs the committed baseline is a regression). Unit-tested with SYNTHETIC findings - zero credits, deterministic. **Live runner** (`runner.py` + `backends.py`): backend-pluggable - reuses Elder's EXACT prompt (`llm_client._build_messages`) + parser (`_parse_response`) so it measures Elder, over a thin OpenAI-compatible transport so each backend (OpenRouter, Poolside, AND sparkles/the-Cave Ollama) is driven independently and scored per-backend. Backend endpoints/keys/models are ALL config-from-env (the Cave/sparkles endpoint is a tailnet address - NEVER a literal in this public repo; supplied via secret at run time). **Baseline** (`baseline.json`): recorded by an on-demand `workflow_dispatch` CI job (`benchmark.sast.yml`) that joins the tailnet (for sparkles) + reads the free-key secrets - never per-PR (no credit burn, no flake in the gate suite). No fabricated verdicts: recall/precision come from real backend runs (AC4). |
| **SAST detection tracer - clear-text-secret-log (#400)** | The first real SAST detection slice (ADR-0006 Option C), end-to-end for ONE class to prove the pipeline before broadening (#401). `personas/code_reviewer/sast.py` (shared): **recall layer** `scan_candidates(hunks) -> tuple[Candidate, ...]` - a PURE builtin detector (a log/print SINK + a secret-ish token on an added line; line numbers computed from the hunk so candidates are diff-anchored). Deliberately LIBERAL: it flags BOTH a real secret-in-log AND the #391 public-config-path log - discriminating them is the judge's job, not the detector's (a detector miss is unrecoverable; an over-flag is recovered by the judge). Per ADR-0006 the heavy Semgrep OSS engine slots in behind this same `scan_candidates` boundary at #401; #400 ships a lean builtin detector to keep the pod footprint flat. **Precision layer** `judge_candidates(...)` - reuses the EXISTING `llm_client.judge_findings` (is_real_bug) as the exploitability gate: candidate -> provisional finding -> judge; KEEP where `is_real_bug` (its `reasoning` becomes the source->sink rationale), SUPPRESS otherwise (the #391 log is dropped with a reason). FAIL-CLOSED on a judge outage / count mismatch (un-triaged candidates are suppressed, never posted raw - the noisy-SAST failure precision prevents). Kept candidates become ordinary `Finding`s; `persona.with_extra_findings` merges them into the evaluation and re-derives the conclusion by the SAME rule `evaluate_diff` uses (high/critical -> failure), so SAST findings flow through the EXISTING anti-hallucination + advisory/blocking publish path - no parallel posting. `dispatch_code_review` runs scan -> judge -> merge after `evaluate_diff`, best-effort (any SAST failure is additive and never breaks the core review). Advisory-by-default; the `code_reviewer_blocking` toggle makes a confirmed leak a merge gate. |
| **SAST full-class coverage - Semgrep engine (#401)** | Broadens detection from the one tracer class to the full set (SQL/command/template injection, SSRF, path-traversal, unsafe-deser, weak-crypto, XXE, hardcoded-credential, cleartext-secret-log) by swapping the ADR-0006 engine in behind the `scan_candidates` boundary #400 built - the ONLY change vs #400, since the judge + `with_extra_findings` merge + publish are engine-agnostic. `scan_candidates(hunks, file_contents, engine=None)` dispatches on `GRUG_SAST_ENGINE` (config value, vendor-neutral): `semgrep` = `scan_semgrep` (the OSS engine), default `builtin` = the #400 zero-dep detector (safe degrade). **scan_semgrep:** writes the changed files (`file_contents`, head-SHA content from #336) to a temp dir, runs `semgrep scan --config services/webhook/sast_rules/ --json` as a CLI subprocess (NO registry network - the rules are VENDORED offline in `sast_rules/python.yml`, one rule per class carrying `metadata.vuln_class`), maps each result to a `Candidate`, and keeps ONLY findings on lines THIS PR added (`DiffHunk.new_lines` per file - a PR review flags what the PR introduces, not pre-existing code). AC5 cost-bound: a per-review byte budget caps what Semgrep scans; over-budget files are skipped + logged (never silent). Best-effort: a missing `semgrep` binary / non-zero exit / timeout / unparseable output returns `()` + logs (SAST is additive). The semgrep CLI ships in the webhook image (`requirements.txt`); `GRUG_SAST_ENGINE=semgrep` is set on the webhook + consumer deployments (the review workloads); the api service never runs the semgrep engine, so it needs neither the dep nor the rules. NOTE (corpus fix): SAST engines allowlist the canonical AWS docs example key, so the #399 hardcoded-credential corpus sample was changed to a realistic non-allowlisted value (else it would measure 0 recall for that class). |
| **SCA - dependency-CVE detection (#434, ADR-0007 Track 1)** | First Track-1 slice: flag dependencies a PR INTRODUCES/bumps-to that have known advisories, as a NEW candidate SOURCE feeding the SAME exploitability-judge + publish pipeline (the engine boundary generalizes from "scan code" to "scan candidates"). `sca.py` (shared): `extract_changed_deps(hunks)` (PURE) parses pinned `name==version` deps on ADDED lines of dependency manifests (requirements*.txt / pyproject / constraints; Python-first, multi-ecosystem deferred), diff-scoped + deduped + capped (AC5); `scan_dependencies(hunks)` queries the free public OSV advisory DB and emits a `Candidate(vuln_class="vulnerable-dependency", file, line, snippet)` per changed dep with advisories. **Engine note:** queries the OSV batch API DIRECTLY over httpx, NOT the pip-audit CLI - pip-audit's requirements-audit mode spins a venv + ensurepip that fails on the pod's readOnlyRootFilesystem (proven: SIGABRT); the direct OSV query is the same advisory data, pod-safe, ONE HTTPS call over the already-allowed 443 egress (free, no key, no SaaS spend, NO new dependency - httpx is already present). `dispatch_code_review` concatenates SAST + SCA candidates into ONE `judge_candidates` pass, then `with_extra_findings` merges the kept ones - so a vulnerable dep is judged for real reachability/impact and published advisory-by-default via the existing path. Best-effort throughout (OSV-unreachable -> () + log). The live OSV test is opt-in (`GRUG_SCA_LIVE_TEST`) so the per-PR gate stays deterministic; mocked tests cover the parse/judge/publish wiring. |
| **Secret/credential scanning (#436, ADR-0007 Track 1)** | Second Track-1 slice: flag committed secrets a PR INTRODUCES on added lines of ANY file type, as a NEW candidate SOURCE feeding the SAME exploitability-judge + publish pipeline. Closes the gap the SAST engine leaves: its vendored ruleset is `languages: [python]`, so a secret committed to a `.env` / YAML / shell / JS / Dockerfile is invisible to it. `secret_scan.py` (shared): `scan_secrets(hunks) -> tuple[Candidate, ...]` is a PURE, file-type-agnostic detector over ADDED lines only (diff-scoped, content-deduped by the exact matched credential per `(file, kind)`, capped at `_MAX_SECRETS=100` and a `_MAX_SCAN_BYTES` aggregate work budget for cost - AC6). Two recall rules: (A) high-signal provider token patterns (AWS access-key id, GitHub / Slack / Google / Stripe tokens, PEM private-key header) matched anywhere on the line, no key-name context needed; (B) a generic secret-ish assignment (`secret`/`token`/`api_key`/`password`/... assigned with `=` or `:`) whose literal value clears a Shannon-entropy gate (length + bits/char). Runtime references and expressions (`credentials?.token`, `process.env.API_KEY`, `$REGISTRY_PASSWORD`, function calls, member access) are not committed values and are rejected before entropy scoring; low-entropy placeholders like `your-key-here` are rejected too. Emits `Candidate(vuln_class="exposed-secret", file, line, snippet)`; `sast._CLASS_LABELS` gains the label. **No-echo invariant:** the snippet MASKS the value (first/last few chars only) so the raw credential is never written into the finding message, the check-run, or the logs; the judge discriminates real-value-vs-placeholder from the full-file context (#336) it already receives, so a known docs example (`AKIA...EXAMPLE`) is SUPPRESSED with a reason while a real leaked key publishes advisory-by-default. `dispatch_code_review` concatenates SAST + SCA + secret candidates into the ONE `judge_candidates` pass; best-effort + additive throughout (a detector failure returns `()` and never breaks the core review). Diff-scoped, so a secret already on an unchanged line is not re-flagged. |
| **IaC misconfiguration scanning (#447, ADR-0007 Track 1)** | Third Track-1 slice (the IaC half of slice 2): flag insecure infrastructure-as-code a PR INTRODUCES on added lines, as another candidate SOURCE feeding the same exploitability-judge + publish pipeline. `iac_scan.py` (shared): `scan_iac(hunks) -> tuple[Candidate, ...]` is pure, diff-scoped, and FILE-TYPE-AWARE - Terraform patterns (`public-read` ACL, `publicly_accessible`, encryption/deletion-protection off) fire only on `*.tf`/`*.tfvars`, Kubernetes/compose patterns (`privileged: true`, `hostNetwork/hostPID/hostIPC`, `runAsNonRoot: false`, `readOnlyRootFilesystem: false`, TLS-verify off) only on YAML, Dockerfile patterns (`USER root`, `:latest` base, `ADD <url>`) only on Dockerfiles; cross-cutting rules (open `0.0.0.0/0` + `::/0`, `chmod 777`, pipe-to-shell) on any IaC file. Cost-bounded like secret_scan (cap 100 + per-line + total-byte budgets), content-deduped per `(file, kind, match)`. Emits `Candidate(vuln_class="iac-misconfig")`; the judge decides exploitability (an intentional public ALB vs an exposed admin port) so kept findings publish advisory-by-default. |
| **Lambda exit complete (#354 / infra#1170+#1172)** | webhook flipped + Lambda/poller/ECR retired 2026-06-12 (live-proven: real redelivered delivery -> pod -> Elder check-run; GitHub deliveries went 500->200 because the route annotation bug shipped fixed on k8s). api flipped via the same SSM upstream override; the k8s poller CronJob unsuspended as the ONLY poller. The CF Worker remains the auth boundary for BOTH hosts; both grug.lol DNS records are hand-owned pending the infra zone migration (the stack token lost zone DNS - infra#239). grug-main DDB has no readers post-api-retirement; table retirement = operator decision. |
| **`parse_diff`** | Pure function in `personas/code_reviewer/diff_parser.py` (shared - services/_shared/, ADR-0014) that takes a unified-diff string and returns `tuple[DiffHunk, ...]`. Hand-rolled (no `unidiff` dep) — the subset we need is small and Lambda cold-start cost of the dep wouldn't pay back. Handles multi-file diffs, skips binary blocks, picks the new-side path on renames. Spec 0015 §Parse contract attests purity. |
| **`DiffHunk`** | Frozen dataclass — fields `file_path: str`, `new_start: int`, `new_lines: frozenset[int]`, `body: str`. `new_lines` is the set of new-file line numbers that were added or are context-with-a-removed-neighbor; the Elder anti-hallucination filter rejects LLM findings whose `(file, line)` is not in this set. `body` retains the raw @@-prefixed text for feeding back to the LLM as review context. |
| **Full-file context (#336)** | The Elder + the judge see the WHOLE current content of each changed file (fetched at head SHA via `GET /contents/{path}?ref=` with the `.raw` Accept header), not just the diff hunk. `dispatch._fetch_file_contents` collects them (capped at `_MAX_CONTEXT_FILES=20`, best-effort — a per-file failure degrades that file to diff-only, never blocks the review); `llm_client._build_messages` / `_build_judge_messages` prepend the 1-based numbered file content (once per path) before the diff, skipping files over `_MAX_FILE_CONTEXT_LINES=800` to bound token cost. The Elder flags ONLY diff additions but READS the whole file, so a mitigation outside the changed lines — a later `finally`/`if: always()` cleanup, a guard above — is visible. Fixes the false-positive class where the cure lives outside the hunk (infra PR #1149: `resource-leak` flagged with an `if: always()` rm 50 lines down; `secret-in-log` flagged for a PATH-not-value env export). Paired prompt wisdom in `code_review_prompt._MITIGATION_SCAN` (scan-before-flag + path-isn't-secret). Backward-compatible: empty `file_contents` ⇒ byte-identical pre-#336 diff-only prompt. |
| **`evaluate_diff`** | Pure function in `personas/code_reviewer/persona.py` (shared) that consumes `tuple[DiffHunk, ...]` + `LlmReviewResponse` (from `llm_client.py`) and produces a `CodeReviewEvaluation`. Drops findings outside the diff (anti-hallucination); maps the wire-format `llm_client.Finding` (`path`, `rule`) to the persona-level `Finding` (`file`, `rule_name`, plus `suggestion`). Spec 0015 §Evaluate contract attests purity. |
| **`Finding` (persona-level)** | Frozen dataclass distinct from `llm_client.Finding`. Fields: `file`, `line`, `severity: Severity`, `rule_name`, `message`, `suggestion`. Posted as GitHub inline review comments by the publisher slice. |
| **`Severity` / `review_types.py`** | `Severity = Literal["low","medium","high","critical"]` + `SEVERITIES = frozenset(get_args(Severity))` live in the shared leaf `review_types.py` (shared - services/_shared/, imports nothing internal) — the SINGLE source (#250). `llm_client`, `persona.py`, and `code_review_prompt.py` all import it (down to the leaf, no cycle), replacing the three independent copies that could silently drift. |
| **`CodeReviewEvaluation`** | Aggregate verdict from one Elder pass. Frozen dataclass — fields `findings: tuple[Finding, ...]`, `passed: bool`, `conclusion: CheckConclusion`. `passed = no high+critical findings`; medium+low are advisory. When the LLM didn't produce content (`kind in {no_diff, all_failed, parse_failed}`), `passed=True` + `conclusion=neutral` — Elder is advisory-first, infra flakiness must not block PRs. Composes 1:1 into a `CheckRunResult` (spec 0001). |
| **`post_review`** | Module-level function in `github_reviews_client.py` (shared) that POSTs a `ReviewResult` to `POST /repos/{owner}/{repo}/pulls/{pull_number}/reviews`. Same pattern as `post_check_run`: no retry/no swallow — `with_install_token_retry` is the caller's responsibility. Spec 0016 §Schema + §Retry contracts. |
| **`ReviewResult`** | Frozen dataclass — fields `commit_id: str`, `event: Literal["COMMENT", "REQUEST_CHANGES"]`, `body: str`, `comments: tuple[InlineComment, ...]`. `event=COMMENT` is the advisory mode; `event=REQUEST_CHANGES` blocks merge. APPROVE + PENDING are intentionally not modeled (an LLM-backed reviewer should not auto-approve, and draft reviews don't publish). `commit_id` pins the review to a specific PR head so it cannot race against new pushes. |
| **`InlineComment`** | Frozen dataclass — fields `path: str`, `line: int` (NEW-side line, matches `DiffHunk.new_lines`), `body: str`. Constructor asserts `line >= 1` because GitHub's PR Reviews API 422s on `line=0`. |
| **`ReviewEvent`** | Literal type — `"COMMENT" \| "REQUEST_CHANGES"`. Restricted subset of GitHub's full enum (APPROVE/COMMENT/REQUEST_CHANGES/PENDING) — see `ReviewResult` for why. |
| **Advisory/blocking gate (`_publish_shape`)** | Single source of truth for the dispatch mode → publish mapping, in `personas/code_reviewer/dispatch.py` (shared). `mode = "blocking" if RepoConfig.code_reviewer_blocking else "advisory"`. `_publish_shape(evaluation, mode)` returns ONE aligned `(check_conclusion, review_event)` pair so the check-run and the inline review can't drift: advisory mode OR any `degraded_reason` → `("neutral", "COMMENT")` (Elder cannot block on infra flakiness); blocking + `conclusion=="failure"` → `("failure", "REQUEST_CHANGES")`; else `(conclusion, "COMMENT")`. Spec 0017 §Gate contract attests this. |
| **Dual-publish independence** | `dispatch_code_review` publishes the check-run (`post_check_run`) and the PR review (`post_review`) on INDEPENDENT surfaces — each in its own try/except; a 5xx on one records a `*_publish_failed` flag and falls through to attempt the other (no early return). `_resolve_result` consults BOTH flags. Every upstream failure (diff fetch, parse, LLM outage, publish 5xx) degrades to advisory-neutral via `_publish_degraded` — `dispatch_code_review` never raises a wire-level exception, so the persona can't 500 the webhook handler. Spec 0017 §Publish-independence contract attests this. |
| **Finding dedup** | On a `synchronize`/`reopened` push the Elder re-reviews the whole diff; `personas/code_reviewer/dedup.py` (shared) stops it re-posting inline comments on unchanged lines. Each inline comment carries a hidden `<!-- grug-rule:NAME -->` marker; `prior_keys_from_comments` parses the PR's existing Grug comments into `(file, line, rule)` keys; `dedup_findings` drops new findings whose key matches. Moved/changed lines get a different key → posted as new. Dedup affects only the inline REVIEW; the check-run summary/conclusion still reflect ALL current findings. Fetch failure → empty key set → post everything (a duplicate beats skipping the review). |
| **Prompt library** | The Elder review system prompt, in `code_review_prompt.py` (shared - `services/_shared/`, a sibling of its `llm_client.py` consumer). `ReviewRule` is a frozen dataclass (name / bug_class / description / bad+good example / default severity); `RULES` is at least 15 bug-class rules seeded from the `/audit` stages; `build_system_prompt(variant: PromptVariant = "v1")` renders them deterministically and appends the `Finding` JSON contract. The shared head/tail has two confidence clauses: `v1` is precision-biased and `v2` is recall-biased. Deep production reviews deliberately pin `v2` for both backends. In `fast` mode, `select_prompt_variant(installation_id)` retains the SSM experiment behavior (`off` gives v1, `split` buckets per install, `all_v2` gives v2). Chief writes after `publish_tpm_evaluation` (`findings_count` = failed BLOCKING plan checks); Elder writes after dual-publish (`findings_count` = surviving findings, `conclusion` = the PUBLISHED conclusion so advisory high/critical reads `warn` not `block`) and on the fetch/parse-degraded path (→ `errored`). |

## Elder cave fallback — SQS airlock (ADR-0005, spec 0018)

When **both** cloud LLM backends fail (`review_diff` → `kind=all_failed`), Elder's owned fallback is **"the Cave"** — the operator's self-hosted, OpenAI-compatible LLM, reachable only over the operator's private network (never publicly exposed). It is reached over an **SQS airlock** so the Cave is never exposed and GitHub credentials never leave AWS. Grug and the Cave never connect; they only ever touch the queues. **Part A (#310) is the AWS side, small diffs inline**; the `grug-cave-connector` worker (#316) and the live cutover (#313) are separate slices. The mechanics of the private connection live in the operator's private infra repo.

| Term | Definition |
|---|---|
| **`grug-cave-jobs` / `grug-cave-results`** | The airlock — two SQS **FIFO** queues (Pulumi). `jobs`: webhook pod -> connector. `results`: connector -> webhook (long-polled by `consumer.py`). FIFO gives free content-dedup + per-install ordering (`MessageGroupId=install_id`); **long polling** (`receive_wait_time_seconds=20`, free-tier guard) + **DLQ + redrive** (`maxReceiveCount` 5 jobs / 3 results → `grug-cave-{jobs,results}-dlq`) + DD monitors (DLQ depth + sustained backlog - on the OWNED `grug.sqs.*` gauges since #379 - and a log-based fallback-fired rate) are the resilience hardening (#312). The producer dedups **head-scoped** (a new push = a distinct fallback — unlike the rerun's head-less dedup). SQS is greenfield in grug here; ADR-0004's rerun queue (#305) reuses this scaffolding (extraction deferred, rule-of-three). |
| **`FallbackJob` / `FallbackResult`** | The cross-repo message contract, now the **shared `spark_cave` envelope** (vendored from infra-public at `services/webhook/spark_cave/`, SHA-pinned via `VENDOR.md`; #454/infra#1610 — do not edit in grug, fix upstream + re-vendor). `FallbackJob{schema_version, persona, principal_id, request_id, payload: inline-or-S3 ref}` with `dedup_key = persona:principal:request`; `FallbackResult{..., ok, result, error}`. grug maps persona=`elder`, principal=install id, request=`repo:pr:head_sha`, and rides its rich fields (`payload_version=2`, diff_ref) inside `payload`. The envelope is persona-generic — a second lane (e.g. an in-house judge, #439) is a schema-free addition. |
| **`DiffRef` codec** (`pack_diff`/`unpack_diff`, #311) | The deep module behind the airlock's diff delivery. A `DiffRef` is a JSON dict that's EITHER `{"kind":"inline","hunks":[…]}` (small PR) OR `{"kind":"s3","bucket":…,"key":…}` (large PR). `pack_diff(hunks)` inlines under `_MAX_INLINE_DIFF_BYTES` (250 KB), else spills the hunks to the `grug-cave-diffs` S3 bucket (`diffs/<install>/<head>.json`, 7-day lifecycle) and returns the pointer; `None` when a big diff can't spill (no bucket / put failed → enqueue no-ops, best-effort). `unpack_diff(diff_ref)` reconstructs the hunks (inline payload or S3 fetch) — the webhook packs, the connector (#316) unpacks, so a big diff is read from S3, **never re-fetched from GitHub** (the creds boundary). IAM: webhook `s3:PutObject` + connector `s3:GetObject`, scoped to `diffs/*`. |
| **`enqueue_fallback` (producer)** | `cave_fallback.enqueue_fallback` (webhook-only). Called from `dispatch_code_review` **only** when `llm_response.kind=="all_failed"` AND `secrets_loader.get_fallback_enabled()` is true (SSM `/grug/elder-fallback-enabled`, fallback-safe → `false`). `SendMessage` to `grug-cave-jobs` with `MessageDeduplicationId` from `(install,repo,pr,persona)` + `MessageGroupId=install_id`. **Best-effort / never raises** — a SendMessage failure logs `elder_fallback_enqueue_failed` and is swallowed (the degraded `errored` verdict was already published; the fallback re-triggers on the next push). The verdict stays `errored` until healed (the "no lies" rule). |
| **`handle_fallback_result` (consumer)** | `cave_fallback.handle_fallback_result` (webhook-only). `consumer.py` long-polls `grug-cave-results.fifo` and feeds each message here wrapped in the `Records` batch shape. Parses a `FallbackResult` (shared `spark_cave.schema`), `post_check_run` with the connector findings, then `record_check_verdict` — which upserts idempotently per `(persona, head_sha)`, **healing `errored` -> `reviewed`** for the current head (a moved-on PR appends a fresh row). **Never raises back to the consumer** (a raise would retry-storm the queue); a malformed/degraded result is logged and dropped. |

## Enforcement concepts

| Term | Definition |
|---|---|
| **GitHub Ruleset** | A GitHub Repository Ruleset that requires specific status checks to pass before merging. Grug creates rulesets to enforce its plan check on the default branch. Managed via the Repository Rulesets API (`POST/GET/DELETE /repos/{owner}/{repo}/rulesets`). See [`services/_shared/github_rulesets_client.py`](services/_shared/github_rulesets_client.py) (shared - services/_shared/, ADR-0014). |
| **Grug-managed ruleset** | A ruleset whose `name` starts with the canonical prefix `Grug - ` (ASCII) or the historical prefix `Grug — ` (pre-cutover). Both prefixes are the ownership marker (`GRUG_RULESET_PREFIXES`): `detect_enforcement()` and the removal fallback use them to distinguish rulesets Grug created from externally-managed ones. |
| **EnforcementState** | Literal type — `"grug_managed" \| "external" \| "none"`. Returned by `detect_enforcement()`. `grug_managed`: at least one Grug-prefixed ruleset (either prefix) exists with `required_status_checks` matching the check name. `external`: no grug-managed ruleset, but the check is enforced via a non-Grug ruleset or legacy branch protection. `none`: check is not enforced anywhere. |
| **`detect_enforcement()`** | Module-level function in `github_rulesets_client.py` that determines the enforcement state for a given status check. Queries both the Rulesets API and the legacy Branch Protection API (`GET /repos/{owner}/{repo}/branches/{branch}/protection/required_status_checks`) to cover repos that haven't migrated to rulesets. **Resilient GETs:** both reads go through `_get_with_retry` — bounded jittered-exponential-backoff retries on GitHub's secondary rate limit (429, rate-limit-403, 5xx) + transport errors, honoring a (capped) `Retry-After`, with `rulesets_get_retry`/`_exhausted` logs. The dashboard fans one `/enforcement` request out per repo in parallel, which tripped GitHub's secondary limit → a false "not enforced"; the retry budget is small (fits the api request budget) and a SUSTAINED limit is handled by the endpoint fallback, not a long wait. |
| **Enforcement detection fallback** | `get_enforcement` (api) wraps live detection: on a rate-limit/transport error that survives the retries, it degrades to the **last-known stored state** rather than 500ing or rendering a false "none" — `grug_managed` if a `enforcement_ruleset_id` is stored, else `unknown` (a distinct `EnforcementState` the badge shows as muted "status unknown", never "⚠ not enforced"). Response carries `degraded: true`. A legitimate `404` (repo not found) is NOT caught (only `httpx` errors degrade). The dashboard hook (`useEnforcement`) additionally retries with **jittered** backoff so its parallel retries don't re-burst. Logged as `enforcement_detect_fallback`. |
| **Legacy branch protection** | Pre-rulesets mechanism for requiring status checks. Still active on many repos. `detect_enforcement()` checks this as a fallback when no ruleset-based enforcement is found. |
| **`enforcement.py`** | Shared module (`services/_shared/`) containing enforcement lifecycle functions. `ensure_enforcement()` detects current state and creates a Grug-managed ruleset if none exists; `remove_enforcement()` deletes it. Both persist the `enforcement_ruleset_id` on the `RepoConfig` store row. See [`services/_shared/enforcement.py`](services/_shared/enforcement.py). |
| **`ensure_enforcement()`** | Idempotent function: detect → skip if `grug_managed` or `external` → create ruleset → store `enforcement_ruleset_id`. Called on installation created (webhook) and persona enable (API toggle). |
| **`remove_enforcement()`** | Deletes the Grug-managed ruleset by ID (read from `RepoConfig`) and clears the stored `enforcement_ruleset_id`. Called on persona disable (API toggle). |
| **`enforcement_ruleset_id`** | Optional integer field on the `RepoConfig` store row. Stores the GitHub ruleset ID of the Grug-managed enforcement ruleset for quick lookup during delete. `None` means no Grug-managed ruleset exists. |
| **`force_disable_enforcement`** | Optional boolean field on the `RepoConfig` store row, default `False`. When `True`, the self-healing loop skips re-creation of a deleted Grug-managed ruleset. Escape hatch for users who intentionally remove enforcement without disabling the TPM persona. |
| **`heal_enforcement()`** | Module-level function in `enforcement.py`. Called from dispatcher when a `repository_ruleset` deleted event fires for a Grug-managed ruleset. Clears the stale `enforcement_ruleset_id`, delegates to `ensure_enforcement()` for idempotent re-creation, and emits an `enforcement_healed` structured log with old + new ruleset IDs. Skipped when `force_disable_enforcement` is `True` or TPM persona is disabled. |
| **Self-healing** | Reconciliation loop: when a Grug-managed ruleset is externally deleted, the webhook re-creates it if the repo still wants enforcement. Triggered by `repository_ruleset` webhook event with `action=deleted`. The `force_disable_enforcement` flag on `RepoConfig` is the opt-out. |
| **Enforcement migration** | One-time backfill script (`scripts/migrate_enforcement.py`) that scans all installations for TPM-enabled repos and creates Grug-managed rulesets where none exist. Handles the grug repo's legacy branch protection → ruleset migration. Supports `--dry-run`. Idempotent. |
| **`grug.enforcement.state`** | DogStatsD gauge metric emitted by `emit_enforcement_metric()` in `observability.py` on every enforcement state change. Value: 1.0 (grug_managed), 0.5 (external), 0.0 (none). Tags: `repo`, `persona`, `enforcement_type`. Used by the enforcement gap monitor. |
| **Enforcement gap monitor** | DD monitor `grug-enforcement-gap` that alerts when any repo has `enforcement_type:none` for >1 hour. Routes to the DD monitoring Discord webhook. |

## Operational concepts

| Term | Definition |
|---|---|
| **Dependency-aware readiness (#404)** | `/readyz` means "this pod can do its job", not just "process up". The `readiness` module (shared) probes the CRITICAL dependencies - SSM+KMS (a SecureString read that exercises AWS auth + KMS decrypt in one call, so a deleted/invalid pod key is caught) and Postgres (`SELECT 1`) - TTL-cached (~5s) and **fail-closed** (any error -> not ready, never falsely ready). The api + webhook `/readyz` handlers return 503 when any dependency is unreachable. Effect: a dependency-broken pod fails readiness, so k8s stops routing to it AND a rollout of broken pods never completes - turning a silent cluster-wide outage (the 2026-06-14 deleted-key incident) into a self-limiting deploy. This "last-good pod keeps serving" property requires the deploy strategy be `RollingUpdate` with `maxUnavailable: 0` (the new pod must pass readiness before the serving pod is removed); api + webhook were switched from `Recreate` to exactly that (#404). The probe is fail-FAST (tight SSM timeouts/retries + a bounded, pool-independent Postgres connection) so a slow dependency or busy pool can't blow the kubelet probe timeout and drop the replica. `/livez` stays a cheap process-up check so a transient dependency blip restarts nothing. Probe `timeoutSeconds` is raised (cache-miss does real I/O). The queue consumer (no HTTP) gets the equivalent via a fail-fast startup self-check (#405, below). |
| **Consumer fail-fast startup self-check (#405)** | The `grug-consumer` process has no HTTP server, so the kubelet has no `/readyz` to gate it on - a pod with a broken AWS key would start its poll threads, hit the receive-error backoff forever, and sit idle "Running" while its queues silently back up. `consumer._startup_check()` runs FIRST in `main()` (before any poll thread spawns) and calls the SAME `readiness.check_readiness()` the `/readyz` handlers use - one definition of "is AWS reachable", no second copy. If any critical dependency is unreachable it logs `consumer_startup_check_failed` and `raise SystemExit(1)`, surfacing as a visible CrashLoopBackOff the kubelet paces. This complements the pre-existing thread watchdog (a poll thread that dies mid-run also exits non-zero); together they cover both "broken before the first poll" and "broke during a poll". |
| **k8s-native alerting (#406)** | The Lambda->k8s migration left the core monitors querying retired `aws.lambda.*` metrics (`errors`/`invocations`/`enhanced.init_duration`), so `[grug-webhook] 5xx`, `[grug-api] 5xx` and `[grug] cold-start p99` sat in permanent `No Data` with `notify_no_data=False` - structurally unable to fire (the 2026-06-14 outage paged nobody). Those three are RETIRED and replaced with signals built on Kubernetes State Metrics, which DO flow for the grug namespace: workload-not-ready (`kubernetes_state.deployment.replicas_ready{kube_namespace:grug} by {kube_deployment} < 1` - covers webhook/api/consumer being down, the availability equivalent of the old 5xx alert), CrashLoopBackOff (`kubernetes_state.container.status_report.count.waiting{kube_namespace:grug,reason:crashloopbackoff} by {pod_name} > 0` - covers grug-consumer, the workload that crash-looped unmonitored), and a restart-spike signal. KSM queries scope by `kube_namespace:grug` (the reliably-present tag), NOT the stack `env` tag (which doesn't propagate onto KSM series - relying on it would reintroduce a `No Data` trap). Folds in #379's poller half: a poller CronJob-health monitor (`kubernetes_state.cronjob.duration_since_last_successful{kube_cronjob:grug-poller}`). #379's queue half shipped later as OWNED telemetry (see the owned queue-depth row): `aws.sqs.*` is not collected by the DD AWS integration in this org, so the three original queue/DLQ monitors sat permanently blind until repointed at the consumer-emitted gauges. Continuous-KSM monitors (workload-not-ready, poller) set `notify_no_data=True` (No Data == the telemetry pipeline broke = page); conditional ones (crashloop, restart-spike) set `notify_no_data=False` (No Data == healthy). Every grug monitor routes to the single `@webhook-grug-discord-monitoring` Discord webhook integration. |
| **Owned queue-depth telemetry (#379)** | The consumer emits `grug.sqs.messages_visible` / `grug.sqs.messages_not_visible` (tag `queue:<name>`, env auto-tagged) every ~60s for ALL six grug queues (both consumed FIFOs, the connector-consumed `grug-cave-jobs.fifo`, and the three DLQs) via `GetQueueAttributes` + the DogStatsD UDP path (`observability.emit_gauge`). Why owned: the DD AWS integration does not collect `aws.sqs.*` in this org, which left the three queue/DLQ monitors PERMANENTLY blind (No Data + `notify_no_data=false`). Age-of-oldest is CloudWatch-only, so backlog monitors use SUSTAINED-depth semantics (`min(last_15m) > 0` = never drained once); DLQ monitors stay any-message (`max > 0`). ONE monitor per queue with a simple `{env:<env>,queue:<name>}` filter - every query env-scoped (gemini peer review: an unscoped query lets a healthy dev emitter mask a dead prod one) and no OR/AND boolean filter syntax (an apply-time validation risk preview cannot catch). Each sweep also emits a per-queue 1/0 `grug.sqs.telemetry_queue_ok` boolean; the `[grug-consumer] Queue telemetry degraded (15min)` monitor (`avg(last_15m):avg ... by {queue} < 0.5`) is the family's ONLY `notify_no_data=true` pager - avg-of-avg so an INTERMITTENT partial failure cannot hide behind one good sweep (codex peer review) while a single throttle blip does not flap; a renamed/vanished queue no-datas its group, and total silence = consumer down. The telemetry thread uses its own tight-timeout SQS client (poolside peer review: default ~60s timeouts could wedge the sweep for minutes). All seven monitors live in `dd_monitors.create_owned_queue_monitors` with a synth test pinning each (query, notify_no_data) pair, plus BIDIRECTIONAL cross-package guard tests (monitor queue tags vs the consumer's emission tuple). Queue URLs are derived (prefix of a known queue-URL env + the fixed Pulumi `name=`s); DLQ reads needed the `ReadDlqDepth` grant on the ra-grug tenant (infrastructure PR #1620). Telemetry thread is daemon + excluded from the consumer watchdog: its failure pages via No Data, never restarts the review-path pod. |
| **Consumer APM writer init (#406)** | grug-consumer emits ZERO APM spans while grug-webhook (same image, same node-local agent) traces fine - and a peer service on the same OCI node traces fine - so it is NOT a node/agent problem (the NetworkPolicy already permits APM egress TCP 8126 and `DD_AGENT_HOST`/`DD_VERSION` are wired). The pre-#405 consumer logged `ddtrace ... failed to start writer service` from a botocore SQS call on a poll thread: the spans that matter are created on worker poll threads, where ddtrace's lazy writer-service start can fail and silently drop spans. Mitigation (`consumer._warm_trace_writer`): open one ddtrace span on the MAIN thread before the poll threads spawn, initializing the writer in the main-thread context. Efficacy is confirmed by checking grug-consumer spans appear in Datadog AFTER deploy; if they still don't, the root cause is elsewhere and tracked separately (not assumed fixed). Fail-safe: telemetry never breaks consumer startup. **#412 follow-up:** the warmup fixed the main-thread startup span but the worker-thread per-poll SQS spans still did not reach Datadog over hours, while a local repro under the same ddtrace (3.19.7) could NOT reproduce a worker-thread bug (worker spans flush fine there) - so the prod cause is environmental, not span-creation. Fix: `consumer._flush_traces()` calls `tracer.flush()` from the MAIN thread (the path proven to deliver) on each watchdog tick, draining the shared buffer so worker-thread spans are delivered regardless of what stalls the long-running periodic writer. |
| **Missed-delivery replay (#407)** | GitHub does not automatically redeliver failed webhooks, so an inline Chief/Hunt Plan failure or durable Elder enqueue failure would leave a required check missing without owned recovery (infra #1254). The App-JWT replay module lists recent webhook deliveries, groups attempts by `guid`, and asks GitHub to redeliver only guids with no 200-399 attempt. The `grug-poller` runs this every 15 minutes; manual replay is also documented in the runbook. A successful redelivery suppresses later attempts, and delivery/snapshot claims protect the relevant async paths from duplicate work. SQS messages themselves use visibility redrive and DLQs rather than this webhook-delivery replay. |
| **Cutover** | Migration path from a self-hosted to a hosted setup, or between Pulumi stacks. Runbook at [`docs/CUTOVER.md`](docs/CUTOVER.md). |
| **HITL prerequisite** | A human-in-the-loop step that must precede automated work (e.g. App registration on github.com, SSM secret pre-load). Documented in [`docs/HITL_PREREQUISITES.md`](docs/HITL_PREREQUISITES.md). |

## Appendix — Slice plan tracking convention (post `/temper-improve`)

SDD slices are planned + executed against the temper specs above. Slice
plans live **in the repo** under
`specs/<NNNN-spec-slug>/plans/<YYYY-MM-DD>-<slice>.md`, not in `/tmp/`
or chat scrollback. Every shipped plan gets an `## Outcome` postscript
capturing the v1→v2 transition (what was proposed, what peer-review
flagged, what shipped). The historical record is the only way future-you
can ask "why does the adapter do X instead of Y?" and get an answer.

**Template:** `specs/SLICE_PLAN_TEMPLATE.md` — copy when starting a new
slice. Bakes the recurring patterns peer-review catches (IOA atomicity,
runtime attester allowlist-not-denylist, cross-user defense at storage
layer, single transactional mutation, clock injection, explicit
deferral checklist, shared-package discipline, storage-side scope of mutation)
as required sections so the v1→v2 ceremonial round collapses to
substantive review.

**Workflow:**
1. Copy `specs/SLICE_PLAN_TEMPLATE.md` to
   `specs/<NNNN>/plans/<date>-<slice>.md`.
2. Fill all required sections (or explicit `N/A — <reason>`).
3. Optional but recommended for first slice on a new IOA action: submit
   to `/peer-review` chain. If BLOCK'd, rewrite to v2 in the same file
   with the v1 narrative + v2 deltas preserved as a transition record.
4. Implement against the plan; spec edits are spec-first (TOML → temper
   verify → grounding attester → code).
5. After commit, add `## Outcome` postscript with shipped SHA +
   reviewer-finding pre-emption matrix.

**When a recurring CRIT class becomes mechanically detectable** (e.g.
the over-broad-DELETE pattern from PR #151 could become an AST-walking
attester that flags `delete_item` calls outside test fixtures), the rule
moves OUT of the template INTO a CI gate or grounding attester. Until
then, template + per-slice peer-review are the human-layer enforcement.

**PII guard:** `services/_shared/tests/test_log_pii_guard.py`
scans for raw secret-bearing field names in log calls (OAuth plaintext
tokens, KMS plaintext keys, App private keys). Does NOT scan for
`github_user_id` / `install_id` — those are intentionally logged with
DD as the authorized sink + support flow needing the raw id. Migrating
identifiers to `observability.fingerprint()` is a deliberate future
call, not a CI blocker today. New log calls referencing the secret-set
field names get red-X'd at PR time unless wrapped with
`observability.fingerprint()` or logging the `_blob` / `_encrypted`
form.
