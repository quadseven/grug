# CONTEXT.md ÔÇö grug domain glossary

The vocabulary used in `services/`, `infra/`, and `web/`. Terms map to identifiers a contributor will encounter while reading code. New contributors should be able to read this file and use the terms correctly without grepping. Drift between this file and the code is a bug; fix the file in the same PR that renames a concept.

> **Architecture decisions live in [`docs/adr/`](docs/adr/).** This file is the lexicon; the ADRs are the load-bearing choices.

## Product surface

| Term | Definition |
|---|---|
| **Grug** | The bot. A hosted GitHub App at `grug.lol` that gates pull requests against a set of process checks. Open-source AGPL-3.0; self-host path in [`docs/SELF_HOST.md`](docs/SELF_HOST.md). |
| **Grug Boss** | Public-facing name of the GitHub App users install on their repos. Same product as Grug; "Grug Boss" is the GitHub Marketplace listing handle. |
| **Persona** | A bounded behavior surface Grug applies per PR. v1 ships exactly one: **TPM** (static Definition-of-Ready check). v1.5+ roadmap (per PRD #21) adds `code-reviewer`, `release-manager`, `stuck-PR-pulse`, and the LLM scope-review half of TPM. Personas are per-repo togglable. |
| **TPM persona** | Today's only persona. Static **Definition-of-Ready (DoR) check** that blocks merge if PR body is malformed. The companion **Scope review** half is wired as a `poolside_client.py` hook in `evaluate()` per the `__init__.py` docstring but is NOT shipped ÔÇö see "Roadmap" rows. See [`services/{api,webhook}/personas/tpm/`](services/api/personas/tpm/). |
| **Elder persona** | The code-reviewer persona. Reads the PR diff and posts inline review comments + a check-run summary. Runs alongside TPM on every `pull_request` event (`opened`, `synchronize`, `ready_for_review`, `reopened`). The two personas dispatch INDEPENDENTLY ÔÇö one failing does not skip the other. v1 modules: `personas/code_reviewer/diff_parser.py` (unified-diff Ôćĺ `DiffHunk`s) + `personas/code_reviewer/persona.py` (`evaluate_diff` rollup) + `personas/code_reviewer/dispatch.py` (orchestration ÔÇö fetch diff, run LLM, publish via reviews+checks clients). LLM-driven via `llm_client.py` (Poolside + OpenRouter round-robin). Advisory by default (`code_reviewer_blocking=False` Ôćĺ check-run `conclusion=neutral`, review `event=COMMENT`); blocking flip via dashboard toggle. See specs 0015 + 0016 + [`services/{api,webhook}/personas/code_reviewer/`](services/api/personas/code_reviewer/). |
| **Pulse (roadmap)** | Scheduled (non-PR-triggered) persona work, named `stuck-PR-pulse` in the future-persona list in `services/webhook/main.py`. Not implemented. Intent: weekly issue-grooming sweep against Grugboard. |

## Process-gate concepts

| Term | Definition |
|---|---|
| **Definition of Ready (DoR)** | The standard a PR description must meet before it can merge. Enforced as a set of `CheckResult`s combined into a single GitHub check-run named `Grug ÔÇö Definition of Ready`. |
| **DoR check** | Individual rule: `why`, `acceptance`, `estimate`, `scope-fence`, `issue-link`. Defined in [`services/{api,webhook}/personas/tpm/dor_checks.py`](services/api/personas/tpm/dor_checks.py) (mirrored ÔÇö see ADR-0001). The five rule names are the prose label ÔÇö there is no `DoRCheck` class; rules are functions returning `CheckResult`. |
| **CheckResult** | Outcome of one `DoR check` against one PR body. Frozen dataclass ÔÇö fields `name: str`, `passed: bool`, `detail: str`. Pass/fail only ÔÇö no third "warn" state. |
| **TpmEvaluation** | Aggregate result of running all `DoR check`s against one PR. Frozen dataclass returned by `evaluate_pull_request(...)` in `personas/tpm/persona.py`. Composes into a `CheckRunResult` for GitHub. |
| **CheckRunResult** | Frozen dataclass that maps directly onto GitHub's Checks API `POST /repos/{owner}/{repo}/check-runs` payload. Carries the `status=completed` Ôćö `conclusion` cross-field invariant ÔÇö enforced in `__post_init__`. See [`services/{api,webhook}/github_checks_client.py`](services/api/github_checks_client.py). |
| **`post_check_run` (publisher)** | Module-level function in `github_checks_client.py` that POSTs a `CheckRunResult` to GitHub. The acceptance-criteria spelling "CheckRunPublisher" is the *concept name* ÔÇö the actual identifier is a function, not a class. |
| **Scope review (roadmap)** | Advisory LLM pass over PR title + body. Wired as a `poolside_client.py` hook called from `evaluate_pull_request(...)` per the `personas/tpm/__init__.py` docstring, but **`poolside_client.py` does not exist in the repo today** ÔÇö feature is roadmap-only. Intended behavior: flag titleÔćöbody mismatch, AC testability, scope-creep, XL inflation; posted as a comment, never blocking. |

## Elder persona ÔÇö code review

| Term | Definition |
|---|---|
| **Async Elder offload (#272)** | The Elder LLM path runs **off** the synchronous webhook ACK. `receive_github_webhook` verifies HMAC, runs TPM (fast, no LLM) inline, then for the code-reviewer persona it **self-invokes** the same webhook Lambda (`async_dispatch.enqueue_elder_review` Ôćĺ `boto3 lambda.invoke(FunctionName=$AWS_LAMBDA_FUNCTION_NAME, InvocationType="Event")`) and returns `result="queued"` ÔÇö so GitHub is ACKed in <10s regardless of the 1ÔÇô2 LLM calls' latency. The async invocation arrives as a raw event (not a Function-URL HTTP event); `lambda_handler.handler` sniffs the `grug_async_job` sentinel and routes to `async_dispatch.run_elder_job` **before** Mangum (which can't parse a non-HTTP event). `run_elder_job` is **idempotent on `delivery_id`** via `install_store.claim_delivery` (an atomic `INSERT .. ON CONFLICT` win-once claim on `PK=DELIVERY#<id>` with a 24h `ttl` + expired-claim takeover) ÔÇö a GitHub redelivery or an AWS async-invoke retry of the same delivery claims-and-skips, so the review is never double-posted. It then calls the unchanged `dispatch_code_review` (its advisory/blocking gate + dual-publish contract from spec 0017 are identical ÔÇö only the *invocation site* moved). `run_elder_job` never re-raises (so AWS doesn't retry-storm; failures degrade to a published neutral check per the existing contract). The shared webhook Lambda `timeout_seconds` is raised to **420s** to cover Elder's worst case (2 backends ├Ś 3 retries ├Ś 30s + judge Ôëł 300s); the sync ACK path returns in ~1ÔÇô2s (its httpx calls are each individually bounded), so the ceiling only ever bites the async path. The execution role gains `lambda:InvokeFunction` on its own ARN (`allow_self_invoke=True` in `lambda_service.create`). Enqueue failure (rare Lambda throttle) logs `elder_enqueue_failed` + returns `result="enqueue_failed"` ÔÇö NO sync fallback, preserving the <10s ACK guarantee (a lost review re-triggers on the next push). **Considered + rejected:** a dedicated worker Lambda (duplicates ~15 env vars + secret grants + a new role + mirror surface ÔÇö the drift ADR-0001 fights) and SQS+worker (queue + event-source-mapping + visibility-vs-Lambda-timeout overhead ÔÇö overkill for grug's volume). |
| **k8s execution paths (#368)** | Off-Lambda the same two async surfaces exist with k8s-native mechanics. **Elder:** `enqueue_elder_review` detects `GRUG_K8S_RUNTIME` (set only in the pod manifests) and runs `run_elder_job` on an in-process **daemon thread** - a pod has no invoke budget to escape, so the ACK handler returns immediately while the thread runs with the pod's lifetime. Idempotency + never-raise live in `run_elder_job`, unchanged. **Trade-off (decided):** a pod restart mid-review drops the in-flight review - the SAME best-effort contract as the Lambda async invoke (a dropped review re-triggers on the next push); rejected SQS+worker for the hot path (new queue + DLQ + IAM + consumer for semantics we already accept). **Queues:** `consumer.py` (new entrypoint, same webhook image, Deployment `grug-consumer`, 1 replica) long-polls `grug-rerun-jobs.fifo` + `grug-cave-results.fifo` one thread each and feeds each message to the SAME handler the ESM used, in the SAME `Records` batch-of-1 shape - handlers and their tests untouched. Delivery semantics preserved: a rerun handler raise leaves the message (visibility-timeout redrive -> DLQ after 3), cave-results always deletes. A dead consumer thread kills the process (watchdog) -> kubelet restart. **IAM:** the `grug-k8s-pod` user already held the three queue grants; #368 adds write-only `s3:PutObject` on the cave-diffs `diffs/*` prefix (Elder spillover now happens on-pod). |
| **Lambda exit complete (#354 / infra#1170+#1172)** | webhook flipped + Lambda/poller/ECR retired 2026-06-12 (live-proven: real redelivered delivery -> pod -> Elder check-run; GitHub deliveries went 500->200 because the route annotation bug shipped fixed on k8s). api flipped via the same SSM upstream override; the k8s poller CronJob unsuspended as the ONLY poller. The CF Worker remains the auth boundary for BOTH hosts; both grug.lol DNS records are hand-owned pending the infra zone migration (the stack token lost zone DNS - infra#239). grug-main DDB has no readers post-api-retirement; table retirement = operator decision. |
| **`parse_diff`** | Pure function in `personas/code_reviewer/diff_parser.py` (mirrored ÔÇö ADR-0001) that takes a unified-diff string and returns `tuple[DiffHunk, ...]`. Hand-rolled (no `unidiff` dep) ÔÇö the subset we need is small and Lambda cold-start cost of the dep wouldn't pay back. Handles multi-file diffs, skips binary blocks, picks the new-side path on renames. Spec 0015 ┬žParse contract attests purity. |
| **`DiffHunk`** | Frozen dataclass ÔÇö fields `file_path: str`, `new_start: int`, `new_lines: frozenset[int]`, `body: str`. `new_lines` is the set of new-file line numbers that were added or are context-with-a-removed-neighbor; the Elder anti-hallucination filter rejects LLM findings whose `(file, line)` is not in this set. `body` retains the raw @@-prefixed text for feeding back to the LLM as review context. |
| **Full-file context (#336)** | The Elder + the judge see the WHOLE current content of each changed file (fetched at head SHA via `GET /contents/{path}?ref=` with the `.raw` Accept header), not just the diff hunk. `dispatch._fetch_file_contents` collects them (capped at `_MAX_CONTEXT_FILES=20`, best-effort ÔÇö a per-file failure degrades that file to diff-only, never blocks the review); `llm_client._build_messages` / `_build_judge_messages` prepend the 1-based numbered file content (once per path) before the diff, skipping files over `_MAX_FILE_CONTEXT_LINES=800` to bound token cost. The Elder flags ONLY diff additions but READS the whole file, so a mitigation outside the changed lines ÔÇö a later `finally`/`if: always()` cleanup, a guard above ÔÇö is visible. Fixes the false-positive class where the cure lives outside the hunk (infra PR #1149: `resource-leak` flagged with an `if: always()` rm 50 lines down; `secret-in-log` flagged for a PATH-not-value env export). Paired prompt wisdom in `code_review_prompt._MITIGATION_SCAN` (scan-before-flag + path-isn't-secret). Backward-compatible: empty `file_contents` Ôçĺ byte-identical pre-#336 diff-only prompt. |
| **`evaluate_diff`** | Pure function in `personas/code_reviewer/persona.py` (mirrored) that consumes `tuple[DiffHunk, ...]` + `LlmReviewResponse` (from `llm_client.py`) and produces a `CodeReviewEvaluation`. Drops findings outside the diff (anti-hallucination); maps the wire-format `llm_client.Finding` (`path`, `rule`) to the persona-level `Finding` (`file`, `rule_name`, plus `suggestion`). Spec 0015 ┬žEvaluate contract attests purity. |
| **`Finding` (persona-level)** | Frozen dataclass distinct from `llm_client.Finding`. Fields: `file`, `line`, `severity: Severity`, `rule_name`, `message`, `suggestion`. Posted as GitHub inline review comments by the publisher slice. |
| **`Severity` / `review_types.py`** | `Severity = Literal["low","medium","high","critical"]` + `SEVERITIES = frozenset(get_args(Severity))` live in the shared leaf `review_types.py` (mirrored, imports nothing internal) ÔÇö the SINGLE source (#250). `llm_client`, `persona.py`, and `code_review_prompt.py` all import it (down to the leaf, no cycle), replacing the three independent copies that could silently drift. |
| **`CodeReviewEvaluation`** | Aggregate verdict from one Elder pass. Frozen dataclass ÔÇö fields `findings: tuple[Finding, ...]`, `passed: bool`, `conclusion: CheckConclusion`. `passed = no high+critical findings`; medium+low are advisory. When the LLM didn't produce content (`kind in {no_diff, all_failed, parse_failed}`), `passed=True` + `conclusion=neutral` ÔÇö Elder is advisory-first, infra flakiness must not block PRs. Composes 1:1 into a `CheckRunResult` (spec 0001). |
| **`post_review`** | Module-level function in `github_reviews_client.py` (mirrored) that POSTs a `ReviewResult` to `POST /repos/{owner}/{repo}/pulls/{pull_number}/reviews`. Same pattern as `post_check_run`: no retry/no swallow ÔÇö `with_install_token_retry` is the caller's responsibility. Spec 0016 ┬žSchema + ┬žRetry contracts. |
| **`ReviewResult`** | Frozen dataclass ÔÇö fields `commit_id: str`, `event: Literal["COMMENT", "REQUEST_CHANGES"]`, `body: str`, `comments: tuple[InlineComment, ...]`. `event=COMMENT` is the advisory mode; `event=REQUEST_CHANGES` blocks merge. APPROVE + PENDING are intentionally not modeled (an LLM-backed reviewer should not auto-approve, and draft reviews don't publish). `commit_id` pins the review to a specific PR head so it cannot race against new pushes. |
| **`InlineComment`** | Frozen dataclass ÔÇö fields `path: str`, `line: int` (NEW-side line, matches `DiffHunk.new_lines`), `body: str`. Constructor asserts `line >= 1` because GitHub's PR Reviews API 422s on `line=0`. |
| **`ReviewEvent`** | Literal type ÔÇö `"COMMENT" \| "REQUEST_CHANGES"`. Restricted subset of GitHub's full enum (APPROVE/COMMENT/REQUEST_CHANGES/PENDING) ÔÇö see `ReviewResult` for why. |
| **Advisory/blocking gate (`_publish_shape`)** | Single source of truth for the dispatch mode Ôćĺ publish mapping, in `personas/code_reviewer/dispatch.py` (mirrored). `mode = "blocking" if RepoConfig.code_reviewer_blocking else "advisory"`. `_publish_shape(evaluation, mode)` returns ONE aligned `(check_conclusion, review_event)` pair so the check-run and the inline review can't drift: advisory mode OR any `degraded_reason` Ôćĺ `("neutral", "COMMENT")` (Elder cannot block on infra flakiness); blocking + `conclusion=="failure"` Ôćĺ `("failure", "REQUEST_CHANGES")`; else `(conclusion, "COMMENT")`. Spec 0017 ┬žGate contract attests this. |
| **Dual-publish independence** | `dispatch_code_review` publishes the check-run (`post_check_run`) and the PR review (`post_review`) on INDEPENDENT surfaces ÔÇö each in its own try/except; a 5xx on one records a `*_publish_failed` flag and falls through to attempt the other (no early return). `_resolve_result` consults BOTH flags. Every upstream failure (diff fetch, parse, LLM outage, publish 5xx) degrades to advisory-neutral via `_publish_degraded` ÔÇö `dispatch_code_review` never raises a wire-level exception, so the persona can't 500 the webhook handler. | **Finding dedup** | On a `synchronize`/`reopened` push the Elder re-reviews the whole diff; `personas/code_reviewer/dedup.py` (mirrored) stops it re-posting inline comments on unchanged lines. Moved/changed lines get a different key Ôćĺ posted as new. Dedup affects only the inline REVIEW; the check-run summary/conclusion still reflect ALL current findings. Fetch failure Ôćĺ empty key set Ôćĺ post everything (a duplicate beats skipping the review). |

| **LLM-as-a-judge** | Quality-feedback loop. ## Activity feed ÔÇö check verdicts (PRD #301, Slice S1)

| Term | Definition |
|---|---|
| **`Check verdict` / `CheckVerdictRecord`** | The recorded outcome of one **Persona**'s check-run on one PR ÔÇö the atom of the Activity feed ("What Grug did"). Persisted by `install_store.put_check_verdict` / read by `list_check_verdicts` (mirrored, ADR-0001). Single-table: `PK=INST#<install_id>`, `SK=ACT#<head_sha>#<persona>`. **Idempotent per `(persona, head_sha)`**: re-reviewing the same commit upserts (heals the row), a new commit appends. Stores RAW facts (`persona` caveman key, `repo`, `pr_number`, `head_sha`, `conclusion`, `summary`, `findings_count`, `blocking`, `created_at`, sparse `degraded_reason`) + a denormalized `verdict` for cheap filtering ÔÇö the raw facts are canonical (ADR-0003). 90-day `ttl` bounds the `ACT#` partition; `list_check_verdicts` sorts by `created_at` desc in SQL (the SK encodes `head_sha`, not time). |
| **`Verdict` / `verdict()`** | The at-a-glance badge ÔÇö `block` / `warn` / `pass` / `errored` ÔÇö derived from a Check verdict's raw facts by the SINGLE pure function `review_types.verdict(conclusion, findings_count, degraded_reason)` (ADR-0003). Precedence: `degraded_reason` set Ôćĺ `errored` (never `pass` ÔÇö "no lies"); `conclusion=="failure"` Ôćĺ `block`; conclusion NOT in {`success`,`neutral`} (cancelled/timed_out/action_required/skipped/stale ÔÇö Grug never concluded) Ôćĺ `errored`; `findings_count != 0` (any non-zero, incl. a defensive negative) Ôćĺ `warn`; else `pass`. Both the write path and the future `/activity` read path call this ONE function; the frontend renders it verbatim (single source of truth). |
| **Persona name mapping (`chief`/`elder`)** | `review_types.persona_for_key` / `key_for_persona` map the legacy code keys (`tpm`/`code_reviewer`) Ôćö the canonical caveman names (`chief`/`elder`, ADR-0002). The ONE boundary the map lives at; `CheckVerdictRecord.persona` stores the caveman name. |
| **`GET /activity` (read endpoint, S2)** | `installations.list_activity` (api service only ÔÇö the SPA's read path, not mirrored): `GET /api/v1/installations/{install_id}/activity`. Session-authed via `_ensure_can_access` (404 unknown install, 403 not-yours). Returns `{activity: [...]}` newest-first. The `verdict` badge is **re-derived server-side** from each row's raw facts via `review_types.verdict` (ADR-0003 ÔÇö NOT the stored denorm, so a mapping change heals history on read); the SPA renders it verbatim, never re-derives. `?verdict=` filters to one badge (fetches a wider window so the cap applies to MATCHING rows); `limit` caps (default 50). The web `ActivityPanel` calls it via the `useActivity` hook, replacing the honest empty state with real rows + the ERRORED badge + ALL/BLOCK/WARN/PASS/ERRORED filter chips. |
| **`record_check_verdict` (best-effort writer)** | `activity_log.record_check_verdict` (mirrored) is the ONE place both dispatchers write a Check verdict: map the persona key, derive the `verdict`, stamp `created_at`, upsert. **Never raises** ÔÇö a recording failure logs `check_verdict_record_failed` and is swallowed so it can't break the check-run the dispatcher just published (same discipline as capture-on-publish + the #272 offload). Chief writes after `publish_tpm_evaluation` (`findings_count` = failed BLOCKING DoR checks); Elder writes after dual-publish (`findings_count` = surviving findings, `conclusion` = the PUBLISHED conclusion so advisory high/critical reads `warn` not `block`) and on the fetch/parse-degraded path (Ôćĺ `errored`). |

## Elder cave fallback ÔÇö SQS airlock (ADR-0005, spec 0018)

When **both** cloud LLM backends fail (`review_diff` Ôćĺ `kind=all_failed`), Elder's owned fallback is **"the Cave"** ÔÇö the operator's self-hosted, OpenAI-compatible LLM, reachable only over the operator's private network (never publicly exposed). It is reached over an **SQS airlock** so the Cave is never exposed and GitHub credentials never leave AWS. Grug and the Cave never connect; they only ever touch the queues. **Part A (#310) is the AWS side, small diffs inline**; the `grug-cave-connector` worker (#316) and the live cutover (#313) are separate slices. The mechanics of the private connection live in the operator's private infra repo.

| Term | Definition |
|---|---|
| **`grug-cave-jobs` / `grug-cave-results`** | The airlock ÔÇö two SQS **FIFO** queues (Pulumi). `jobs`: webhook Lambda Ôćĺ connector. `results`: connector Ôćĺ webhook Lambda (consumed via SQS event-source mapping). FIFO gives free content-dedup + per-install ordering (`MessageGroupId=install_id`); **long polling** (`receive_wait_time_seconds=20`, free-tier guard) + **DLQ + redrive** (`maxReceiveCount` 5 jobs / 3 results Ôćĺ `grug-cave-{jobs,results}-dlq`) + DD monitors (DLQ depth, queue age, fallback-fired rate) are the resilience hardening (#312). The producer dedups **head-scoped** (a new push = a distinct fallback ÔÇö unlike the rerun's head-less dedup). SQS is greenfield in grug here; ADR-0004's rerun queue (#305) reuses this scaffolding (extraction deferred, rule-of-three). |
| **`FallbackJob` / `FallbackResult`** | The cross-repo message contract (webhook-only dataclasses, NOT mirrored ÔÇö like `async_dispatch`). Schema **v2** (#311): `FallbackJob` carries install/repo/pr/head coords + a **`diff_ref`** (the DiffRef codec's output) and **no GitHub credential**. `FallbackResult` carries the connector's findings + a `degraded` marker. The connector consumes the job and produces the result; it never calls GitHub. |
| **`DiffRef` codec** (`pack_diff`/`unpack_diff`, #311) | The deep module behind the airlock's diff delivery. A `DiffRef` is a JSON dict that's EITHER `{"kind":"inline","hunks":[ÔÇŽ]}` (small PR) OR `{"kind":"s3","bucket":ÔÇŽ,"key":ÔÇŽ}` (large PR). `pack_diff(hunks)` inlines under `_MAX_INLINE_DIFF_BYTES` (250 KB), else spills the hunks to the `grug-cave-diffs` S3 bucket (`diffs/<install>/<head>.json`, 7-day lifecycle) and returns the pointer; `None` when a big diff can't spill (no bucket / put failed Ôćĺ enqueue no-ops, best-effort). `unpack_diff(diff_ref)` reconstructs the hunks (inline payload or S3 fetch) ÔÇö the webhook packs, the connector (#316) unpacks, so a big diff is read from S3, **never re-fetched from GitHub** (the creds boundary). IAM: webhook `s3:PutObject` + connector `s3:GetObject`, scoped to `diffs/*`. |
| **`enqueue_fallback` (producer)** | `cave_fallback.enqueue_fallback` (webhook-only). Called from `dispatch_code_review` **only** when `llm_response.kind=="all_failed"` AND `secrets_loader.get_fallback_enabled()` is true (SSM `/grug/elder-fallback-enabled`, fallback-safe Ôćĺ `false`). `SendMessage` to `grug-cave-jobs` with `MessageDeduplicationId` from `(install,repo,pr,persona)` + `MessageGroupId=install_id`. **Best-effort / never raises** ÔÇö a SendMessage failure logs `elder_fallback_enqueue_failed` and is swallowed (the degraded `errored` verdict was already published; the fallback re-triggers on the next push). The verdict stays `errored` until healed (the "no lies" rule). |
| **`handle_fallback_result` (consumer)** | `cave_fallback.handle_fallback_result` (webhook-only). `lambda_handler.handler` routes the `aws:sqs` event (a `Records` list with `eventSource=="aws:sqs"`) to it **before** Mangum (which can't parse a non-HTTP event), alongside the existing `grug_async_job` sentinel. Parses a `FallbackResult`, `post_check_run` with the connector findings, then `record_check_verdict` ÔÇö which upserts idempotently per `(persona, head_sha)`, **healing `errored` Ôćĺ `reviewed`** for the current head (a moved-on PR appends a fresh row). **Never raises back to the Lambda** (a raise would retry-storm the event-source mapping); a malformed/degraded result is logged and dropped. |

## Enforcement concepts

| Term | Definition |
|---|---|
| **GitHub Ruleset** | A GitHub Repository Ruleset that requires specific status checks to pass before merging. Grug creates rulesets to enforce its DoR check on the default branch. Managed via the Repository Rulesets API (`POST/GET/DELETE /repos/{owner}/{repo}/rulesets`). See [`services/{api,webhook}/github_rulesets_client.py`](services/api/github_rulesets_client.py) (mirrored ÔÇö see ADR-0001). |
| **Grug-managed ruleset** | A ruleset whose `name` starts with the prefix `Grug ÔÇö`. This prefix is the ownership marker: `detect_enforcement()` uses it to distinguish rulesets Grug created from externally-managed ones. |
| **EnforcementState** | Literal type ÔÇö `"grug_managed" \| "external" \| "none"`. Returned by `detect_enforcement()`. `grug_managed`: at least one `Grug ÔÇö`-prefixed ruleset exists with `required_status_checks` matching the check name. `external`: no grug-managed ruleset, but the check is enforced via a non-Grug ruleset or legacy branch protection. `none`: check is not enforced anywhere. |
| **`detect_enforcement()`** | Module-level function in `github_rulesets_client.py` that determines the enforcement state for a given status check. Queries both the Rulesets API and the legacy Branch Protection API (`GET /repos/{owner}/{repo}/branches/{branch}/protection/required_status_checks`) to cover repos that haven't migrated to rulesets. **Resilient GETs:** both reads go through `_get_with_retry` ÔÇö bounded jittered-exponential-backoff retries on GitHub's secondary rate limit (429, rate-limit-403, 5xx) + transport errors, honoring a (capped) `Retry-After`, with `rulesets_get_retry`/`_exhausted` logs. The dashboard fans one `/enforcement` request out per repo in parallel, which tripped GitHub's secondary limit Ôćĺ a false "not enforced"; the retry budget is small (fits the 15s api Lambda) and a SUSTAINED limit is handled by the endpoint fallback, not a long wait. |
| **Enforcement detection fallback** | `get_enforcement` (api) wraps live detection: on a rate-limit/transport error that survives the retries, it degrades to the **last-known stored state** rather than 500ing or rendering a false "none" ÔÇö `grug_managed` if a `enforcement_ruleset_id` is stored, else `unknown` (a distinct `EnforcementState` the badge shows as muted "status unknown", never "ÔÜá not enforced"). Response carries `degraded: true`. A legitimate `404` (repo not found) is NOT caught (only `httpx` errors degrade). The dashboard hook (`useEnforcement`) additionally retries with **jittered** backoff so its parallel retries don't re-burst. Logged as `enforcement_detect_fallback`. |
| **Legacy branch protection** | Pre-rulesets mechanism for requiring status checks. Still active on many repos. `detect_enforcement()` checks this as a fallback when no ruleset-based enforcement is found. |
| **`enforcement.py`** | Mirrored module containing enforcement lifecycle functions. `ensure_enforcement()` detects current state and creates a Grug-managed ruleset if none exists; `remove_enforcement()` deletes it. Both persist the `enforcement_ruleset_id` on the `RepoConfig` store row. See [`services/{api,webhook}/enforcement.py`](services/api/enforcement.py). |
| **`ensure_enforcement()`** | Idempotent function: detect Ôćĺ skip if `grug_managed` or `external` Ôćĺ create ruleset Ôćĺ store `enforcement_ruleset_id`. Called on installation created (webhook) and persona enable (API toggle). |
| **`remove_enforcement()`** | Deletes the Grug-managed ruleset by ID (read from `RepoConfig`) and clears the stored `enforcement_ruleset_id`. Called on persona disable (API toggle). |
| **`enforcement_ruleset_id`** | Optional integer field on the `RepoConfig` store row. Stores the GitHub ruleset ID of the Grug-managed enforcement ruleset for quick lookup during delete. `None` means no Grug-managed ruleset exists. |
| **`force_disable_enforcement`** | Optional boolean field on the `RepoConfig` store row, default `False`. When `True`, the self-healing loop skips re-creation of a deleted Grug-managed ruleset. Escape hatch for users who intentionally remove enforcement without disabling the TPM persona. |
| **`heal_enforcement()`** | Module-level function in `enforcement.py`. Called from dispatcher when a `repository_ruleset` deleted event fires for a Grug-managed ruleset. Clears the stale `enforcement_ruleset_id`, delegates to `ensure_enforcement()` for idempotent re-creation, and emits an `enforcement_healed` structured log with old + new ruleset IDs. Skipped when `force_disable_enforcement` is `True` or TPM persona is disabled. |
| **Self-healing** | Reconciliation loop: when a Grug-managed ruleset is externally deleted, the webhook re-creates it if the repo still wants enforcement. Triggered by `repository_ruleset` webhook event with `action=deleted`. The `force_disable_enforcement` flag on `RepoConfig` is the opt-out. |
| **Enforcement migration** | One-time backfill script (`scripts/migrate_enforcement.py`) that scans all installations for TPM-enabled repos and creates Grug-managed rulesets where none exist. Handles the grug repo's legacy branch protection Ôćĺ ruleset migration. Supports `--dry-run`. Idempotent. |
| **`grug.enforcement.state`** | DogStatsD gauge metric emitted by `emit_enforcement_metric()` in `observability.py` on every enforcement state change. Value: 1.0 (grug_managed), 0.5 (external), 0.0 (none). Tags: `repo`, `persona`, `enforcement_type`. Used by the enforcement gap monitor. |
| **Enforcement gap monitor** | DD monitor `grug-enforcement-gap` that alerts when any repo has `enforcement_type:none` for >1 hour. Routes to the DD monitoring Discord webhook. |

## Operational concepts

| Term | Definition |
|---|---|
| **Dependency-aware readiness (#404)** | `/readyz` means "this pod can do its job", not just "process up". The `readiness` module (mirrored) probes the CRITICAL dependencies - SSM+KMS (a SecureString read that exercises AWS auth + KMS decrypt in one call, so a deleted/invalid pod key is caught) and Postgres (`SELECT 1`) - TTL-cached (~5s) and **fail-closed** (any error -> not ready, never falsely ready). The api + webhook `/readyz` handlers return 503 when any dependency is unreachable. Effect: a dependency-broken pod fails readiness, so k8s stops routing to it AND a rollout of broken pods never completes - turning a silent cluster-wide outage (the 2026-06-14 deleted-key incident) into a self-limiting deploy. This "last-good pod keeps serving" property requires the deploy strategy be `RollingUpdate` with `maxUnavailable: 0` (the new pod must pass readiness before the serving pod is removed); api + webhook were switched from `Recreate` to exactly that (#404). The probe is fail-FAST (tight SSM timeouts/retries + a bounded, pool-independent Postgres connection) so a slow dependency or busy pool can't blow the kubelet probe timeout and drop the replica. `/livez` stays a cheap process-up check so a transient dependency blip restarts nothing. Probe `timeoutSeconds` is raised (cache-miss does real I/O). The queue consumer (no HTTP) gets the equivalent via a fail-fast startup self-check (#405, below). |
| **Consumer fail-fast startup self-check (#405)** | The `grug-consumer` process has no HTTP server, so the kubelet has no `/readyz` to gate it on - a pod with a broken AWS key would start its poll threads, hit the receive-error backoff forever, and sit idle "Running" while its queues silently back up. `consumer._startup_check()` runs FIRST in `main()` (before any poll thread spawns) and calls the SAME `readiness.check_readiness()` the `/readyz` handlers use - one definition of "is AWS reachable", no second copy. If any critical dependency is unreachable it logs `consumer_startup_check_failed` and `raise SystemExit(1)`, surfacing as a visible CrashLoopBackOff the kubelet paces. This complements the pre-existing thread watchdog (a poll thread that dies mid-run also exits non-zero); together they cover both "broken before the first poll" and "broke during a poll". |
| **k8s-native alerting (#406)** | The Lambda->k8s migration left the core monitors querying retired `aws.lambda.*` metrics (`errors`/`invocations`/`enhanced.init_duration`), so `[grug-webhook] 5xx`, `[grug-api] 5xx` and `[grug] cold-start p99` sat in permanent `No Data` with `notify_no_data=False` - structurally unable to fire (the 2026-06-14 outage paged nobody). Those three are RETIRED and replaced with signals built on Kubernetes State Metrics, which DO flow for the grug namespace: workload-not-ready (`kubernetes_state.deployment.replicas_ready{kube_namespace:grug} by {kube_deployment} < 1` - covers webhook/api/consumer being down, the availability equivalent of the old 5xx alert), CrashLoopBackOff (`kubernetes_state.container.status_report.count.waiting{kube_namespace:grug,reason:crashloopbackoff} by {pod_name} > 0` - covers grug-consumer, the workload that crash-looped unmonitored), and a restart-spike signal. KSM queries scope by `kube_namespace:grug` (the reliably-present tag), NOT the stack `env` tag (which doesn't propagate onto KSM series - relying on it would reintroduce a `No Data` trap). Folds in #379's poller half: a poller CronJob-health monitor (`kubernetes_state.cronjob.duration_since_last_successful{kube_cronjob:grug-poller}`). #379's consumer queue-age half is DEFERRED: `aws.sqs.*` is not collected by the DD AWS integration in this org, so a queue-age monitor would be permanent `No Data` - the exact trap this slice retires. The consumer's crash/down outage mode is covered by the crashloop + workload-not-ready monitors; a true queue-age signal needs the SQS integration namespace enabled first. Continuous-KSM monitors (workload-not-ready, poller) set `notify_no_data=True` (No Data == the telemetry pipeline broke = page); conditional ones (crashloop, restart-spike) set `notify_no_data=False` (No Data == healthy). Every grug monitor routes to the single `@webhook-grug-discord-monitoring` Discord webhook integration. |
| **Consumer APM writer init (#406)** | grug-consumer emits ZERO APM spans while grug-webhook (same image, same node-local agent) traces fine - and a peer service on the same OCI node traces fine - so it is NOT a node/agent problem (the NetworkPolicy already permits APM egress TCP 8126 and `DD_AGENT_HOST`/`DD_VERSION` are wired). The pre-#405 consumer logged `ddtrace ... failed to start writer service` from a botocore SQS call on a poll thread: the spans that matter are created on worker poll threads, where ddtrace's lazy writer-service start can fail and silently drop spans. Mitigation (`consumer._warm_trace_writer`): open one ddtrace span on the MAIN thread before the poll threads spawn, initializing the writer in the main-thread context. Efficacy is confirmed by checking grug-consumer spans appear in Datadog AFTER deploy; if they still don't, the root cause is elsewhere and tracked separately (not assumed fixed). Fail-safe: telemetry never breaks consumer startup. |
| **Missed-delivery replay (#407)** | The DoR/TPM check runs INLINE on the webhook, so a GitHub delivery that arrives while grug is down is lost - GitHub retries a few times then gives up, leaving a required check stuck until a human re-triggers (infra #1254). Recovery uses the GitHub App webhook-deliveries API (App-JWT, no installation token): `GET /app/hook/deliveries` lists recent deliveries (cursor-paginated, newest first; a delivery is SUCCESS when `status_code` is 200-399, FAILED at 400-599 or a connection error). The deep module `delivery_replay` groups deliveries by `guid` (shared across an event's original + its redeliveries) and `POST /app/hook/deliveries/{id}/attempts` re-sends only guids where NO delivery succeeded - so a guid whose redelivery later succeeds is skipped on the next pass (idempotent WITHOUT persistence, and the window is bounded so even if guid-sharing ever changed, the worst case is a redundant redeliver, never a duplicate check - `post_check_run` is idempotent on (name, head_sha)). Auto-recovery: the `grug-poller` CronJob (every 15m) calls `replay_since(window)` best-effort each run, so missed deliveries self-replay with no recovery-detection plumbing. Manual fallback: `scripts/replay_deliveries.py --since T` (runbook-documented). The async SQS layer (grug-rerun-jobs / grug-cave-results) is already resilient - messages persist with a visibility-timeout+DLQ and the consumer auto-drains on recovery (#368/#405) - so #407 adds NOTHING there beyond a documented note. |

| **HITL prerequisite** | A human-in-the-loop step that must precede automated work (e.g. App registration on github.com, SSM secret pre-load). Documented in [`docs/HITL_PREREQUISITES.md`](docs/HITL_PREREQUISITES.md). |

## Appendix ÔÇö Slice plan tracking convention (post `/temper-improve`)

SDD slices are planned + executed against the temper specs above. Slice
plans live **in the repo** under
`specs/<NNNN-spec-slug>/plans/<YYYY-MM-DD>-<slice>.md`, not in `/tmp/`
or chat scrollback. Every shipped plan gets an `## Outcome` postscript
capturing the v1Ôćĺv2 transition (what was proposed, what peer-review
flagged, what shipped). The historical record is the only way future-you
can ask "why does the adapter do X instead of Y?" and get an answer.

**Template:** `specs/SLICE_PLAN_TEMPLATE.md` ÔÇö copy when starting a new
slice. Bakes the recurring patterns peer-review catches (IOA atomicity,
runtime attester allowlist-not-denylist, cross-user defense at storage
layer, single transactional mutation, clock injection, explicit
deferral checklist, mirror discipline, storage-side scope of mutation)
as required sections so the v1Ôćĺv2 ceremonial round collapses to
substantive review.

**Workflow:**
1. Copy `specs/SLICE_PLAN_TEMPLATE.md` to
   `specs/<NNNN>/plans/<date>-<slice>.md`.
2. Fill all required sections (or explicit `N/A ÔÇö <reason>`).
3. Optional but recommended for first slice on a new IOA action: submit
   to `/peer-review` chain. If BLOCK'd, rewrite to v2 in the same file
   with the v1 narrative + v2 deltas preserved as a transition record.
4. Implement against the plan; spec edits are spec-first (TOML Ôćĺ temper
   verify Ôćĺ grounding attester Ôćĺ code).
5. After commit, add `## Outcome` postscript with shipped SHA +
   reviewer-finding pre-emption matrix.

**When a recurring CRIT class becomes mechanically detectable** (e.g.
the over-broad-DELETE pattern from PR #151 could become an AST-walking
attester that flags `delete_item` calls outside test fixtures), the rule
moves OUT of the template INTO a CI gate or grounding attester. Until
then, template + per-slice peer-review are the human-layer enforcement.

**PII guard:** `services/{api,webhook}/tests/test_log_pii_guard.py`
scans for raw secret-bearing field names in log calls (OAuth plaintext
tokens, KMS plaintext keys, App private keys). Does NOT scan for
`github_user_id` / `install_id` ÔÇö those are intentionally logged with
DD as the authorized sink + support flow needing the raw id. Migrating
identifiers to `observability.fingerprint()` is a deliberate future
call, not a CI blocker today. New log calls referencing the secret-set
field names get red-X'd at PR time unless wrapped with
`observability.fingerprint()` or logging the `_blob` / `_encrypted`
form.
